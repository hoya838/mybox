지배 관계 — 채점표가 나머지 전부를 다스린다

도메인지식이 좋은지도, 워크플로우가 나아졌는지도, 루프가 제대로 도는지도 전부 채점기준이 있어야 알 수 있다. 5요소 중 우선순위 하나만 꼽으면 평가표 정교화. 실제로 이 팀에서 가장 얇은 층이기도 하다 — golden_set은 아직 "기대노트 1개 적중" 수준이며, 규모가 커지면 검증 루브릭화가 재검토 1순위다.

횡단 관심사 — 보안·거버넌스는 층이 아니라 전 층을 가로지른다

도입 3대 걱정 중 하나가 보안이고(AI강사 노트), 에이전트가 아니라 데이터와 거버넌스를 조직이 꽉 잡아야 한다(에이전트4천개 노트). 이 팀에선 판정층의 secret_leak 측정, 구조층의 guard_vault.sh·sensitivity 3단계, 분리 4트리거 1번(보안)이 각 층에 심어진 형태 — 별도 층이 아니라 모든 층의 통과 조건이다.

연결고리 — 노트가 증명하는 층간 상관관계

볼트의 실제 노트 2개 이상이 교차 검증한 관계만 실었다

재료→판정

도메인지식이 곧 채점 능력이다. 변호사 에이전트의 답을 법률 모르는 사람은 검증 못 한다(AI강사 노트) = 심사위원 AI에게 가르치는 건 지식이 아니라 전문가의 "취향"이다(채점회사 노트). 같은 명제의 양면.

판정→루프

채점기준 없으면 루프가 안 돈다. 테스트 스위트가 있어야 에이전트가 통과할 때까지 반복할 수 있고, 없으면 깨진 코드를 완료라 선언한다(에이전틱 5개념). RL도 채점표 없으면 학습 자체가 정지(채점회사). 리워드 해킹 때문에 채점자는 계속 정교해져야 한다 — 수요가 안 마르는 이유.

구조→루프

승인게이트가 스텝 경계에 있어야 루프가 안전하다. 위임 스펙트럼(완전자율에서 인간주도까지)의 판단 기준은 "결과를 빠르게 검증할 수 있는가"(에이전틱 5개념). 이 팀의 vocab_queue·insight_queue·guard_vault.sh가 그 경계 장치 — 구조를 하나로 뭉치면 이 체크포인트가 사라진다.

재료→구조

재료 성격이 구조를 결정한다. 결정론적 재료(순위·수치·API 데이터)는 자동화 파이프에, 확률론적 판단(해석·전략)은 사람 게이트 뒤에 배치(이커머스 자동화). 이 팀 파이프라인도 동형 — 어휘검증·frontmatter는 결정론, 라우팅 판정·인사이트 흡수는 확률론.

루프→구조

루프가 없으면 구조가 늘어나기만 한다. 에이전트 4천개 = 유지보수 4천건(에이전트4천개 노트). 환류(자가점검·통합점검)가 구조 증식을 억제한다 — 신규 에이전트 전에 기존 역할 흡수 검토(self_auditor #11). 저빈도 4개 에이전트 통합점검으로 "안 늘려도 됨"을 확인한 것도 이 루프의 산물.

판정→재료

채점표가 재료 수집을 역규정한다. 선별 3기준(반복·결정·연결)이 볼트에 뭐가 들어올지 정하고, golden_set 커버리지가 어느 부서 지식이 얇은지 드러낸다(balance.md 편중 경보 → 수집 권고). 판정이 입구를 통제하는 구조.

조직→루프

조직이 안 받치면 루프가 멈춘다 — PO의 저주. 전담팀·리소스·중간보고 없이 개인 시간에 떠넘기면 효율이 올라가도 일만 늘어나 의지를 잃는다(AI강사 노트). 기술 루프가 아무리 잘 설계돼도 그걸 돌릴 사람의 시간과 보상이 조직에서 안 나오면 정지 — 도입 실패의 최다 원인이 이 연결의 단절.

하네스 설계 예시 — 회계·인사·PPT·카드뉴스를 하나의 업무OS로

오늘 인사이트를 설계 결정으로 번역한 가상 예시. 업무마다 에이전트를 만들지 않는다 — 범용 에이전트 하나가 스킬·규칙·지식으로 4개 업무를 수행한다

설계 결정 4가지 — 인사이트에서 직행

1. 업무별 에이전트 금지, 스킬로 구분(에이전트4천개 = 유지보수4천건) · 2. 업무마다 채점표를 먼저 정의(채점기준 없으면 개선 불가) · 3. 결정론은 스크립트, 확률론은 승인게이트 뒤(이커머스 자동화) · 4. run_log·승인큐 SLA로 중간보고 대체(PO의 저주 회피)

회계 월마감 — 결정론/확률론 분리 + 채점표 내장

데이터 수집은 결정론이라 스크립트, 증감 해석 코멘트는 확률론이라 초안까지만. 채점표가 스킬 안에 있어 validator가 바로 실행.

# /회계마감 [월]

## 절차
1. [결정론] bin/pull_ledger.py — 은행·카드·세금계산서 데이터 수집
2. [결정론] 계정과목별 집계 → 전월 대비 증감표
3. [확률론] 증감 원인 코멘트 초안 작성 — 지식/2_areas 회계 노트 인용
4. [게이트] 사람 검토 → 승인 후 산출물/보고서/[월]_마감.md 확정

## 채점표 (validator 실행)
- [ ] 원장 합계 = 은행 잔액 대사 일치 (불일치 시 FAIL — 원인 행 표시)
- [ ] 전 계정과목 증감률 표기 누락 0
- [ ] 코멘트에 인용 노트 링크 1개 이상
- [ ] sensitivity: secret 항목(급여 상세) 보고서 혼입 0

업무별 위임 스펙트럼

기준은 하나 — 결과를 빠르게 검증할 수 있는가. 검증 쉬우면 자율, 되돌리기 어려우면 게이트

업무위임 단계이유
인사 온보딩완전자율체크리스트 = 결정론, 결과 검증 즉시 가능. 계정 발급 완료 여부가 곧 채점.

카드뉴스감독위임제작은 자율, 발행(외부공개)은 사람. 나간 게시물은 되돌리기 어려움.
회계 마감협업+게이트수집은 자율이지만 확정·원장수정은 승인 필수. guard_ledger.sh 훅이 강제.

이 설계가 회피하는 실패 3가지

유지보수 지옥 — 업무 10개로 늘어도 에이전트는 그대로, 스킬 10개(각각 독립 갱신) · PO의 저주 — run_log가 자동 중간보고, 승인큐 SLA 7일이 방치 방지 · 저울 없는 다이어트 — 모든 스킬에 채점표 내장, validator가 PASS 없이 완료 선언 차단.

설계 시스템 만들기 — 채점표·루프·워크플로우를 찍어내는 메타-하네스

분야(도메인)를 입력하면 그 분야의 채점표·루프·워크플로우 파일 세트가 나오는 시스템. 하네스를 만드는 하네스 — 이미 보유한 메타팀빌더·하네스팀생성기가 이 계열이다

핵심 순서 — 채점표가 먼저, 워크플로우가 나중

보통은 워크플로우부터 그리고 채점을 뒤에 붙인다. 거꾸로 해야 한다. "이 업무가 잘 됐는지 어떻게 아는가?"에 답 못 하면 워크플로우를 설계할 자격이 없다(저울 없는 다이어트). 채점표가 정해지면 워크플로우는 그 채점표를 통과하는 최단 경로로 자연히 그려지고, 루프는 그 채점을 반복 실행하는 장치일 뿐이다.

5단계 설계 파이프라인

각 단계가 지식지도의 층과 1:1 대응 — 재료를 캐고, 판정을 세우고, 구조를 그리고, 루프를 심고, 조직에 안착

1단계 — 도메인 인터뷰 (1층 재료)

전문가의 머릿속에서 채점 기준의 원석을 캔다

채점표는 발명하는 게 아니라 도메인 전문가가 이미 갖고 있는 "취향"을 명문화하는 것(채점회사 노트 — 전문가 취향 데이터가 금값). 코딩 몰라도 됨, 업무를 알면 됨.

## 설계 질문지 (업무 하나당)
1. 이 업무의 결과물을 보고 3초 안에 "잘됐다/망했다" 판정하는 기준은?
2. 신입이 이 업무를 하면 가장 자주 틀리는 지점 3개는?
3. 절대 일어나면 안 되는 사고 1개는? (→ FAIL 조건·훅 후보)
4. 이 업무를 다시 꺼내 쓰는 순간은 언제인가? (→ 골든질의 후보)
5. 판단이 갈리는 회색지대는? (→ 확률론 = 사람 게이트 후보)
2단계 — 채점표 도출 (4층 판정)

인터뷰 답변을 3종 채점 도구로 변환

질문1·2 → 체크리스트, 질문3 → FAIL 조건, 질문4 → 골든질의. 업무마다 이 3종 세트가 나오면 설계의 절반이 끝난 것.

## 채점표 템플릿 (스킬마다 내장)
- 체크리스트: [ ] 항목별 통과 기준 (전문가 질문1·2에서)
- FAIL 조건: 하나라도 걸리면 즉시 중단+보고 (질문3에서)
- 골든질의: "질문 → 기대 결과물" 쌍, 회귀 테스트용 (질문4에서)
- 측정치: hit_rate·오염·누출 → metrics.json (숫자로 추세 관리)
3단계 — 워크플로우 설계 (2층 구조)

채점표를 통과하는 최단 경로를 그린다

인터뷰 질문5(회색지대)가 경로를 가른다 — 결정론 구간은 스크립트, 확률론 구간은 초안+승인게이트(이커머스 자동화). 업무는 에이전트가 아니라 스킬로(에이전트4천개).

## 분류 기준 → 배치
반복적 + 기준명확 + 저보안  → 완전자율 스킬 (예: 온보딩 체크리스트)
반복적 + 기준명확 + 외부공개 → 감독위임 (제작 자율, 발행 승인)
판단개입 + 되돌리기어려움   → 협업+훅 강제 (예: 원장수정 차단)
감각적 판단 (브랜딩·전략)    → 자동화 제외, AI는 재료 준비만
4단계 — 루프 심기 (3층 유지)

채점을 반복 실행하는 표준 장치 4개를 붙인다

루프는 분야와 무관하게 표준 세트가 동일 — 이 팀에서 검증된 4장치를 그대로 이식하면 된다. PO의저주 회피 장치이기도 하다.

## 표준 루프 세트 (복붙 가능)
1. run_log.jsonl   — 실행마다 append = 자동 중간보고
2. 승인큐 (SLA 7일) — 확률론 구간의 사람 게이트, 방치 시 경보
3. 주기점검        — stale·중복·드리프트 분기 스캔 (decay)
4. 자가점검        — 구조·정합·미실행 스킬 정적 검사 (self_audit)
   + 신규 확장 게이트: "기존이 흡수 가능한가?" 먼저
5단계 — 파일럿 검증 (0층 조직)

전체를 만들기 전에 업무 1개로 2주 돌린다

"만들기 전에 팔아보라"(54명 0원 노트)의 시스템 버전 — 채점표 통과율이 기준선을 넘을 때만 다음 업무로 확장. 전 업무 일괄 구축은 4천개 에이전트의 길.

## 파일럿 게이트
- 업무 1개 선정: 임팩트 크고 + 결정론 비중 높은 것부터
- 2주 운영 → 채점표 통과율·hit_rate 측정
- 기준선 통과 → 다음 업무 추가 (스킬 1개 = 행 1개)
- 미달 → 채점표부터 재점검 (워크플로우 탓하기 전에)

채점표를 못 쓰겠다 — 전문가도 판정 기준을 말로 못 하면 그 업무는 자동화 대상이 아니라 재료 수집 대상 · 스킬이 아니라 에이전트가 늘고 있다 — 흡수 게이트가 안 돌고 있다는 증거 · 파일럿 없이 전 업무 동시 구축 중 — 4천개 에이전트의 길, 즉시 1개로 축소.

웹 개발 — 클릭 하나가 지나가는 다섯 정거장

볼트의 개발 노트 18건을 하나의 여정으로 재구성. 프론트엔드·백엔드·서버는 과목이 아니라 요청 하나가 순서대로 통과하는 정거장 — 왜 생겼는지 알면 외울 필요가 없다

1 프론트엔드

사용자가 보는 화면 — 브라우저

HTML(뼈대)·CSS(옷)·JS(움직임)에서 출발해 화면이 복잡해지자 React(컴포넌트·상태)가, React의 CSR 단점(첫 로딩 느림·SEO 약함)을 풀려고 Next.js(SSR·풀스택)가 나왔다. 모든 기술은 앞 기술의 문제를 풀려고 태어난 결과물 — 계보로 이해하면 외울 게 없다.

2 API 계약

프론트와 백엔드의 약속

API는 프론트와 백엔드가 데이터를 주고받는 약속. REST가 그 규격이고, 일을 CRUD(생성·조회·수정·삭제) 네 동사로 나눠 HTTP 메서드에 매핑한다. 서버가 HTML을 만들어 내려주던 시대에서 JSON 데이터만 반환하는 시대로 — 이 경계선이 프론트/백 분업의 기준선이다.

3 백엔드

요청을 처리하는 본체 — 변하지 않는 심장

백엔드는 "사용자마다 다른 결과를 그 자리에서 만들어주려고" 태어났다(방명록·검색이 계기). 프레임워크는 20년간 계속 바뀌었지만(ASP→MVC→MSA·비동기) 하는 일은 동일 — 인증·검증·라우팅 → 비즈니스 로직·DB연동 → 응답·에러처리·로깅. 도구가 아니라 이 골격을 배우는 것.

백엔드는 왜 생겼나 백엔드 A to Z 종합개론 AI시대 백엔드 방향 신입 백엔드 최소기준

핵심 개념 · 요청-응답 사이클 · 인증/권한 · 비즈니스 로직 · MVC → MSA

데이터를 읽고 쓴다
4 데이터

DB와 스토리지 — 기억을 담당

DB=집, 테이블=방, API=배달부. 구조화 데이터(DB)와 파일(스토리지)은 원래 별개인데 슈퍼베이스 같은 서비스가 인증과 함께 묶어버렸다 — 바이브코딩이 안 되는 최다 원인이 이 층의 기초 부재. 변수(진동벨, 껐다 켜면 사라짐)와 DB(냉장고, 남음)의 구분부터.

바이브코딩 DB 테이블 API 실습 여기까지 알면 풀스택 자료구조 시리즈

핵심 개념 · 테이블·SQL · DB vs 스토리지 vs 변수 · 자료구조(효율의 뿌리)

돌아갈 집을 제공한다
5 서버·인프라

세상에 공개하는 기반 — 배포

서버=그냥 컴퓨터, IP=주소, 도메인=이름, DNS=변환기. 코드 완성에서 인터넷 공개까지가 하나의 흐름이고, 그 컴퓨터를 사지 않고 빌려 쓰는 게 클라우드(IaaS·PaaS·SaaS). 규모가 커지면 로드밸런서·캐시·MSA로 분산 — 로컬에서 도는 것과 서비스는 배포가 가른다.

웹서버 배포 도메인 DNS 클라우드 구성도 기초 트위터 팬아웃 시스템디자인

핵심 개념 · IP·도메인·DNS · 클라우드 3형태 · 로드밸런서 · 배포 vs 로컬

지배 원리 — 도구는 바뀌어도 본질은 안 바뀐다

다섯 정거장의 골격(요청-응답, CRUD, 인증, 데이터 구분, 배포)은 20년간 한 번도 안 바뀌었다. 바뀐 건 언어와 프레임워크뿐. 그래서 투자 순서는 신기술이 아니라 변하지 않는 것부터 — 아키텍처·보안·성능·네트워크·OS. 언어 하나는 사용법이 아니라 동작원리까지 파면, 나머지는 그 원리에서 크게 벗어나지 않는다(AI시대 백엔드 방향 노트).

연결고리 — 정거장 사이에서 태어난 기술들

각 관계는 볼트 노트 2개 이상이 교차 검증한 것만 실었다

프론트엔드↔백엔드

경계선은 계속 움직인다. 처음엔 서버가 HTML을 다 만들어 내려줬고(SSR, 서버사이드스크립트 시대), React가 그걸 브라우저로 가져갔다가(CSR), 첫 로딩·SEO 문제로 Next.js가 다시 서버로 되돌렸다(SSR 부활+풀스택). 두 층의 분업 비율 자체가 기술사의 주요 축.

API→백엔드

약속이 구조를 강제한다. REST·CRUD 규격이 정해지자 백엔드는 "API만 노출하고 JSON만 반환"하는 형태로 수렴했고, 신입 최소기준 1번이 REST API 설계(게시판 CRUD)가 됐다. 계약이 먼저, 구현이 나중.

백엔드→데이터

백엔드가 태어난 이유가 곧 데이터다. "사용자마다 다른 결과"를 만들려면 기억(방명록 글·검색 대상)이 필요했고, 그 기억이 DB다. 백엔드 발전사(CGI→PHP→DB→MVC)에서 DB 연동이 들어온 순간이 정적 웹과 서비스의 분기점.

데이터→인프라

데이터가 커지면 구조가 인프라로 넘어간다. 트위터 팬아웃이 대표 사례 — 팔로워 피드를 읽을 때 조인할지(쓰기 간단) 쓸 때 캐시에 밀어넣을지(읽기 간단)는 DB 설계가 아니라 인프라 트레이드오프이고, 인플루언서 문제로 하이브리드가 답이 됐다.

인프라→프론트엔드

배포가 있어야 서비스다. 로컬에서 도는 앱은 내 컴퓨터 안의 연습이고, IP·도메인·DNS·호스팅을 거쳐야 남이 쓴다. 비개발자가 바이브코딩을 포기하는 5개념 중 1번이 배포/로컬 구분 — 여정의 끝이 다시 사용자 화면으로 이어진다.

백엔드→인프라

규모가 아키텍처를 부른다. 모놀리스로 충분하던 서비스가 커지면 로드밸런서·캐시·MSA로 쪼개진다 — 단 MSA는 유행이 아니라 필요(트래픽·팀 규모)가 근거여야 한다. "과한 쪼갬의 역설"은 에이전트 설계에서도 똑같이 재발견됐다.

횡단 관심사 — AI 시대, 사람 몫은 판단과 책임

AI가 코드는 사람보다 잘 짠다. 하지만 설계·판단·책임은 못 맡긴다 — 어느 정거장을 쪼갤지, 팬아웃을 push로 갈지 pull로 갈지 같은 트레이드오프 결정이 사람 몫이고, 그래서 AI가 코딩을 대체할수록 시스템 디자인 역량의 가치는 오히려 오른다. 다섯 정거장의 원리 이해가 그 판단의 재료다.

자바스크립트 공부 순서 — 개념이 태어난 이유대로 배운다

외우지 않는 학습법: 각 문법은 앞 단계의 불편을 풀려고 나왔다. 그 순서 그대로 8단계 — 웹개발 지도의 1정거장(프론트엔드)을 파고드는 로드맵

학습 원칙 — 왜 생겼는지 알면 외울 필요가 없다

프론트 역사 노트의 핵심: 모든 기술은 문제해결의 결과물. 그리고 OpenAI 연구원 노트의 실행법: AI에게 코드를 시키되 나온 코드는 한 줄 한 줄 읽고, "이 줄은 왜 이렇게 돼?"를 하루 백 번 묻기 — 아래 8단계는 그 질문에 답이 나오는 순서로 배열했다.

1단계 — 변수: 데이터를 담는 상자

let · const

프로그램의 첫 필요 = 값을 기억하기. 변수는 진동벨(끄면 사라짐), 나중에 배울 DB는 냉장고(남음) — 바이브코딩 필수개념 노트의 비유. const(안 바뀜)를 기본으로, 바뀔 것만 let.

const name = "준호";   // 다시 안 바뀌는 값
let count = 0;         // 바뀔 값
count = count + 1;     // OK
// name = "다른이름";  // 에러 — const는 재할당 금지
2단계 — 자료형: 상자에 담기는 값의 종류

문자열 · 숫자 · 불리언 · null/undefined

값마다 성격이 다르다 — 문자는 붙이고, 숫자는 더하고, 불리언은 갈림길 재료. JS는 타입에 관대해서 3 + "5" = "35" 같은 사고가 남 — 이 관대함이 나중에 TS를 배우는 이유가 된다(복선).

const title = "지식지도";  // 문자열 (따옴표)
const notes = 79;          // 숫자
const done = true;         // 불리언 (참/거짓)
let next = null;           // "비어있음"을 일부러 표시
let tbd;                   // undefined — 아직 아무것도 안 넣음
3단계 — 조건문: 갈림길

if · else · 비교연산자

"상황에 따라 다르게" — 백엔드가 태어난 이유(사용자마다 다른 결과)의 문법 버전. 비교는 ===(타입까지 같음)를 쓰는 습관 — ==는 JS의 관대함 때문에 함정.

const hitRate = 1.0;
if (hitRate >= 0.8) {
  console.log("기준선 통과");   // 채점표 그 자체
} else {
  console.log("분리 트리거 검토");
}
4단계 — 배열과 객체: 값의 묶음

[ ] 배열 · { } 객체

변수 하나로는 노트 79개를 못 담는다. 배열=순서 있는 목록(방 번호로 찾기), 객체=이름표 붙은 서랍(이름으로 찾기). DB의 테이블·행 개념이 여기서 시작 — DB=집, 테이블=방 비유의 문법 원형.

const tags = ["회의록", "리서치", "매뉴얼"];  // 배열
tags[0];                    // "회의록" — 순서(0부터)로 꺼냄

const note = {              // 객체
  title: "채점만 해주는 회사",
  department: "프로덕트",
  hit: true
};
note.title;                 // 이름으로 꺼냄
5단계 — 반복문: 같은 일을 여러 번

for · for...of

배열이 생기면 "전부에 대해 똑같이" 하고 싶어진다 — 우리 /일괄 파이프라인이 하는 일이 정확히 이것(inbox 항목마다 같은 처리).

const queue = ["노트A", "노트B", "노트C"];
for (const item of queue) {
  console.log(item + " 정제 완료");  // 항목마다 반복
}
6단계 — 함수: 이름 붙인 레시피

function · 매개변수 · return · 화살표 함수

반복문으로도 안 풀리는 것 = "이 묶음 로직을 다른 곳에서도 쓰고 싶다". 함수는 자판기 — 넣는 것(매개변수)과 나오는 것(return)만 알면 내부는 몰라도 쓴다. API를 이해하는 문법적 기초(API=함수를 네트워크 너머로 부르는 약속).

function judge(hitRate, baseline) {
  return hitRate >= baseline;      // 넣으면 → 나온다
}
judge(1.0, 0.8);                   // true

const judge2 = (h, b) => h >= b;   // 같은 것의 짧은 표기(화살표)
7단계 — DOM과 이벤트: JS가 태어난 이유

querySelector · addEventListener

JS는 원래 "화면을 움직이려고" 태어났다(프론트 역사 노트 — HTML 뼈대·CSS 옷·JS 움직임). 버튼을 찾아서(DOM), 눌리면(이벤트), 반응한다. 이 페이지의 탭 전환도 정확히 이 코드다.

const btn = document.querySelector(".tab-btn"); // 화면에서 찾기
btn.addEventListener("click", () => {           // 눌리면
  console.log("탭 전환");                        // 반응
});
8단계 — 비동기: 기다리는 동안 멈추지 않기

fetch · async/await

서버에 데이터를 요청하면 시간이 걸린다 — 그동안 화면이 얼면 안 됨. 주문하고 진동벨 받고 다른 일 하는 것이 비동기. 여기가 프론트가 백엔드(API)와 만나는 지점 — 웹개발 지도 2정거장으로 이어진다.

async function loadNotes() {
  const res = await fetch("/api/notes");  // 서버에 주문, 올 때까지 대기
  const data = await res.json();          // JSON으로 풀기
  console.log(data);
}

바이브코딩 시대에 왜 문법을 배우나

AI가 코드를 다 짜주는데 왜? — "지름길은 기초를 건너뛰는 게 아니라 기초를 빨리 이해하는 지름길이어야 한다"(OpenAI 연구원 노트). AI가 짠 코드를 한 줄 한 줄 읽을 수 있어야 고칠 수 있고, 오픈북 시험(AI)은 응용력 있는 사람만 잘 본다(AI강사 노트). 이 8단계는 시험공부가 아니라 AI가 낸 답을 검증하는 눈을 만드는 과정.

타입스크립트 공부 순서 — JS의 사고가 곧 커리큘럼

TS는 새 언어가 아니라 JS에 채점표를 얹은 것. JS 8단계를 마친 사람이 겪는 사고 순서대로 8단계 — 각 단계는 "JS에서 이런 사고가 났다 → TS는 이렇게 막는다"

학습 원칙 — 타입은 코드의 채점표다

타입 검사는 점수가 아니라 바이너리 판정(맞는 타입이다/아니다) — 흔들리지 않는 채점의 문법 버전이고, 컴파일러(tsc)는 코드의 validator다. JS의 관대함이 배울 땐 편하지만 프로젝트가 커지고 팀원이 늘면 "어디서 에러 났는지 모르는" 지옥이 되고(TS 노트), 그 지옥의 각 장면이 아래 8단계의 입구다.

1단계 — 왜 TS인가: JS의 관대함이 낸 사고

동기 이해

JS는 3 + "5"를 에러 없이 "35"로 만들고, 숫자 변수에 문자를 넣어도 조용하다. 혼자 할 땐 편한데 협업에선 "누가 뭘 넣었는지" 추적 불가 — 규모 있는 회사가 대부분 TS를 쓰는 이유. TS를 먼저 배울 필요는 없다, JS가 익숙해진 뒤가 적기(TS 노트).

// JS — 전부 무사통과 (그게 문제)
let a = 3 + "5";   // "35" ?!
let b = 42;
b = "asdf";        // 에러 없음. 3주 뒤 어딘가에서 터짐
2단계 — 기본 타입 주석: 상자에 라벨 붙이기

: number · : string · : boolean

변수 뒤에 : 타입 한 줄이 전부 — 이게 타입스크립트의 사실상 전부다(TS 노트 원문). 잘못된 값은 실행 전, 코드 쓰는 순간 빨간 줄로 차단.

let count: number = 3;
let title: string = "지식지도";
let done: boolean = false;

count = "많이";   // 에러: Type 'string' is not assignable to type 'number'
                  // 3주 뒤가 아니라 지금 잡힘
3단계 — 유니온 타입: 짜장 아니면 짬뽕

number | string

"둘 중 뭐가 올지 모른다"고 다 열어버리면(any) 의미가 없다. "짜장도 짬뽕도 좋지만 탕수육은 안 먹어" — 허용 범위를 좁게 명시하는 게 유니온(TS 노트 비유).

let d: number | string;   // 이 둘만 허용
d = 42;        // OK
d = "42번";    // OK
d = null;      // 에러 — "널은 얘기가 없었잖아"
4단계 — 배열과 객체 타입: 묶음에도 라벨

string[] · interface

JS 4단계(배열·객체)의 TS 버전. 배열은 타입[], 객체는 interface로 모양(어떤 이름표에 어떤 타입)을 먼저 선언 — 우리 볼트의 frontmatter 고정 스키마와 같은 발상이다: 모양을 강제하면 드리프트가 없다.

const tags: string[] = ["회의록", "리서치"];
tags.push(3);   // 에러 — 숫자는 못 들어옴

interface Note {          // 노트의 "고정 스키마"
  title: string;
  department: string[];
  hit: boolean;
}
const note: Note = { title: "채점 회사", department: ["프로덕트"], hit: true };
5단계 — 함수 타입: 자판기의 투입구와 배출구

매개변수 타입 + 반환 타입

JS 함수는 아무거나 넣고 아무거나 나올 수 있다. TS는 투입구(매개변수)와 배출구(return)에 전부 라벨 — 함수 서명만 읽으면 내부를 안 봐도 쓸 수 있게 된다. API 명세의 문법 원형.

function judge(hitRate: number, baseline: number): boolean {
  return hitRate >= baseline;
}
judge(1.0, 0.8);      // true
judge("높음", 0.8);   // 에러 — 투입구부터 차단
6단계 — any의 유혹과 금지

any는 최후의 수단

any는 "모든 타입 허용" = JS로 돌아가기. any를 남발하면 TS 쓰는 의미가 없고 팀에서 지적받는다(TS 노트 — "야 any 뭐야 당장 지워"). 정말 미정일 때만, 그것도 잠깐.

let c: any = 4;
c = "문자";      // 무사통과 — 채점표를 스스로 버린 것

// any가 필요해 보이면 대부분 유니온이나 unknown으로 해결됨
let e: unknown = 외부에서온값;   // any보다 안전 — 쓰기 전 타입 확인 강제
7단계 — 타입 추론과 좁히기: TS가 알아서 아는 것

추론 · typeof 좁히기

모든 곳에 타입을 쓸 필요는 없다 — 값을 보면 TS가 추론한다. 유니온처럼 여러 타입일 땐 typeof로 갈림길을 만들면 그 블록 안에서 타입이 좁혀진다(조건문 3단계의 재활용).

let x = 10;        // 안 써도 number로 추론됨
// x = "십";       // 에러 — 추론도 채점표다

function print(v: number | string) {
  if (typeof v === "string") {
    v.toUpperCase();   // 이 블록에선 string 확정 — 문자열 메서드 OK
  } else {
    v.toFixed(1);      // 여기선 number 확정
  }
}
8단계 — 컴파일: 브라우저는 JS만 이해한다

tsc · tsconfig.json

TS는 인간 편의용 — 브라우저는 JS만 읽는다. tsc가 타입 표기를 벗겨 JS로 변환(그 순간 채점 완료), tsconfig.json이 변환 설정. JS 8단계의 배포 개념과 여기서 만난다.

// 터미널
npx tsc            // index.ts → index.js (타입 표기 제거됨)

// tsconfig.json 핵심 3개
{
  "compilerOptions": {
    "outDir": "./dist",     // 변환된 JS를 모을 폴더
    "target": "es6",        // 어떤 버전 JS로 변환할지
    "module": "commonjs"    // 모듈 방식 (React면 달라짐)
  }
}

TS와 이 팀의 구조는 같은 사상이다

interface = frontmatter 고정 스키마, 유니온 = 통제어휘(허용값 목록), tsc = validator, any 금지 = 어휘 자동추가 금지. "모양을 먼저 선언하고 벗어나면 차단한다"는 발상이 코드에선 TS, 지식에선 이 팀의 vocab_guardian이다 — 드리프트 차단이라는 같은 문제의 두 해법.
