2. 연결고리 — 노트가 증명하는 층간 상관관계

볼트의 실제 노트 2개 이상이 교차 검증한 관계만 실었다.

재료→판정

도메인지식이 곧 채점 능력이다. 변호사 에이전트의 답을 법률 모르는 사람은 검증 못 한다 = 심사위원 AI에게 가르치는 건 지식이 아니라 전문가의 "취향"이다. 같은 명제의 양면.

판정→루프

채점기준 없으면 루프가 안 돈다. 테스트 스위트가 있어야 에이전트가 통과할 때까지 반복할 수 있고, 없으면 깨진 코드를 완료라 선언한다. RL도 채점표 없으면 학습 자체가 정지. 리워드 해킹 때문에 채점자는 계속 정교해져야 한다.

구조→루프

승인게이트가 스텝 경계에 있어야 루프가 안전하다. 위임 스펙트럼의 판단 기준은 "결과를 빠르게 검증할 수 있는가". 이 팀의 vocab_queue·insight_queue·guard_vault.sh가 그 경계 장치 — 구조를 하나로 뭉치면 이 체크포인트가 사라진다.

재료→구조

재료 성격이 구조를 결정한다. 결정론적 재료(순위·수치·API)는 자동화 파이프에, 확률론적 판단(해석·전략)은 사람 게이트 뒤에. 이 팀 파이프라인도 동형 — 어휘검증·frontmatter는 결정론, 라우팅 판정·인사이트 흡수는 확률론.

루프→구조

루프가 없으면 구조가 늘어나기만 한다. 에이전트 4천개 = 유지보수 4천건. 환류(자가점검)가 구조 증식을 억제한다 — 신규 에이전트 전에 기존 역할 흡수 검토. 저빈도 4개 에이전트 통합점검으로 "안 늘려도 됨"을 확인한 것도 이 루프의 산물.

판정→재료

채점표가 재료 수집을 역규정한다. 선별 3기준(반복·결정·연결)이 볼트에 뭐가 들어올지 정하고, golden_set 커버리지가 어느 부서 지식이 얇은지 드러낸다. 판정이 입구를 통제하는 구조.

조직→루프

조직이 안 받치면 루프가 멈춘다 — PO의 저주. 전담팀·리소스·중간보고 없이 개인 시간에 떠넘기면 효율이 올라가도 일만 늘어나 의지를 잃는다. 기술 루프가 아무리 잘 설계돼도 그걸 돌릴 사람의 시간과 보상이 조직에서 안 나오면 정지 — 도입 실패의 최다 원인.

3. 하네스 설계 예시 — 4개 업무를 하나의 업무OS로

오늘 인사이트를 설계 결정으로 번역한 가상 예시. 업무마다 에이전트를 만들지 않는다 — 범용 에이전트 하나가 스킬·규칙·지식으로 회계·인사·PPT·카드뉴스 4개 업무를 수행한다.

설계 결정 4가지 — 인사이트에서 직행

1. 업무별 에이전트 금지, 스킬로 구분(에이전트4천개=유지보수4천건) · 2. 업무마다 채점표를 먼저 정의(채점기준 없으면 개선 불가) · 3. 결정론은 스크립트, 확률론은 승인게이트 뒤(이커머스 자동화) · 4. run_log·승인큐 SLA로 중간보고 대체(PO의 저주 회피)

업무별 위임 스펙트럼

기준은 하나 — 결과를 빠르게 검증할 수 있는가. 검증 쉬우면 자율, 되돌리기 어려우면 게이트.

업무위임 단계이유
인사 온보딩완전자율체크리스트 = 결정론, 결과 검증 즉시 가능. 계정 발급 완료 여부가 곧 채점.
PPT 제작완전자율산출물이 파일이라 되돌리기 자유. 채점표(슬라이드 수·인용·토큰준수)로 즉시 검증.
카드뉴스감독위임제작은 자율, 발행(외부공개)은 사람. 나간 게시물은 되돌리기 어려움.
회계 마감협업+게이트수집은 자율이지만 확정·원장수정은 승인 필수. guard_ledger.sh 훅이 강제.

이 설계가 회피하는 실패 3가지

유지보수 지옥 — 업무 10개로 늘어도 에이전트는 그대로, 스킬 10개(각각 독립 갱신) · PO의 저주 — run_log가 자동 중간보고, 승인큐 SLA 7일이 방치 방지 · 저울 없는 다이어트 — 모든 스킬에 채점표 내장, validator가 PASS 없이 완료 선언 차단.

4. 설계 시스템 만들기 — 하네스를 찍어내는 메타-하네스

분야(도메인)를 입력하면 그 분야의 채점표·루프·워크플로우 파일 세트가 나오는 시스템. 하네스를 만드는 하네스.

핵심 순서 — 채점표가 먼저, 워크플로우가 나중

보통은 워크플로우부터 그리고 채점을 뒤에 붙인다. 거꾸로 해야 한다. "이 업무가 잘 됐는지 어떻게 아는가?"에 답 못 하면 워크플로우를 설계할 자격이 없다. 채점표가 정해지면 워크플로우는 그 채점표를 통과하는 최단 경로로 자연히 그려지고, 루프는 그 채점을 반복 실행하는 장치일 뿐이다.

5단계 설계 파이프라인

각 단계가 지식지도의 층과 1:1 대응 — 재료를 캐고, 판정을 세우고, 구조를 그리고, 루프를 심고, 조직에 안착.

1단계 — 도메인 인터뷰 (1층 재료)

전문가의 머릿속에서 채점 기준의 원석을 캔다

채점표는 발명하는 게 아니라 도메인 전문가가 이미 갖고 있는 "취향"을 명문화하는 것. 코딩 몰라도 됨, 업무를 알면 됨.

## 설계 질문지 (업무 하나당)
1. 이 업무의 결과물을 보고 3초 안에 "잘됐다/망했다" 판정 기준은?
2. 신입이 이 업무를 하면 가장 자주 틀리는 지점 3개는?
3. 절대 일어나면 안 되는 사고 1개는? (→ FAIL 조건·훅 후보)
4. 이 업무를 다시 꺼내 쓰는 순간은 언제인가? (→ 골든질의 후보)
5. 판단이 갈리는 회색지대는? (→ 확률론 = 사람 게이트 후보)
2단계 — 채점표 도출 (4층 판정)

인터뷰 답변을 3종 채점 도구로 변환

질문1·2 → 체크리스트, 질문3 → FAIL 조건, 질문4 → 골든질의. 업무마다 이 3종 세트가 나오면 설계의 절반이 끝난 것.

## 채점표 템플릿 (스킬마다 내장)
- 체크리스트: [ ] 항목별 통과 기준 (전문가 질문1·2)
- FAIL 조건: 하나라도 걸리면 즉시 중단+보고 (질문3)
- 골든질의: "질문 → 기대 결과물" 쌍, 회귀 테스트용 (질문4)
- 측정치: hit_rate·오염·누출 → metrics.json (숫자로 추세)
3단계 — 워크플로우 설계 (2층 구조)

채점표를 통과하는 최단 경로를 그린다

질문5(회색지대)가 경로를 가른다 — 결정론 구간은 스크립트, 확률론 구간은 초안+승인게이트. 업무는 에이전트가 아니라 스킬로.

## 분류 기준 → 배치
반복 + 기준명확 + 저보안   → 완전자율 스킬 (온보딩 체크리스트)
반복 + 기준명확 + 외부공개  → 감독위임 (제작 자율, 발행 승인)
판단개입 + 되돌리기어려움  → 협업+훅 강제 (원장수정 차단)
감각적 판단 (브랜딩·전략)   → 자동화 제외, AI는 재료 준비만
4단계 — 루프 심기 (3층 유지)

채점을 반복 실행하는 표준 장치 4개를 붙인다

루프는 분야와 무관하게 표준 세트가 동일 — 이 팀에서 검증된 4장치를 그대로 이식하면 된다.

## 표준 루프 세트 (복붙 가능)
1. run_log.jsonl   — 실행마다 append = 자동 중간보고
2. 승인큐 (SLA 7일) — 확률론 구간 사람 게이트, 방치 시 경보
3. 주기점검        — stale·중복·드리프트 분기 스캔 (decay)
4. 자가점검        — 구조·정합·미실행 스킬 정적 검사 (self_audit)
   + 신규 확장 게이트: "기존이 흡수 가능한가?" 먼저
5단계 — 파일럿 검증 (0층 조직)

전체를 만들기 전에 업무 1개로 2주 돌린다

"만들기 전에 팔아보라"의 시스템 버전 — 채점표 통과율이 기준선을 넘을 때만 다음 업무로 확장. 전 업무 일괄 구축은 4천개 에이전트의 길.

## 파일럿 게이트
- 업무 1개 선정: 임팩트 크고 + 결정론 비중 높은 것부터
- 2주 운영 → 채점표 통과율·hit_rate 측정
- 기준선 통과 → 다음 업무 추가 (스킬 1개 = 행 1개)
- 미달 → 채점표부터 재점검 (워크플로우 탓하기 전에)

실패 신호 3가지 — 이럴 때 멈추고 재설계

채점표를 못 쓰겠다 — 전문가도 판정 기준을 말로 못 하면 그 업무는 자동화 대상이 아니라 재료 수집 대상 · 스킬이 아니라 에이전트가 늘고 있다 — 흡수 게이트가 안 돌고 있다는 증거 · 파일럿 없이 전 업무 동시 구축 중 — 4천개 에이전트의 길, 즉시 1개로 축소.

5. 이 팀의 구현 — 5층을 llm-wiki-team이 실제로 어떻게 만들었나

지식지도의 각 층을 이 팀의 실제 파일·규칙·에이전트로 대응. 층마다 만든 것 → 하는 기능 순.
