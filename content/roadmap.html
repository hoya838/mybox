개발자 로드맵 — 입문부터 심화까지

"코딩 한 번도 안 해봤다" "실무 개발자"까지 순서대로. 단계마다 뭘·왜 배우는지.

핵심 원칙: 외우지 말고 "왜 생겼는지" 흐름으로 이해. 모든 기술은 문제 해결의 결과다. 그리고 입력만 말고 직접 만들어봐야 실력이 된다. → 이 태도를 방법론으로 정리한 학습법(거꾸로 배우기)을 먼저 읽으면 이 로드맵을 어떤 순서로 볼지 잡힌다.

레벨 0 — 개념 잡기 입문

1
코딩 vs 프로그래밍
프로그램 = 컴퓨터가 할 일을 순서대로 적은 것(운동 프로그램처럼). 코딩 = 컴퓨터에 일 시키기. 코드 = 그 수단.
쉽게: 컴퓨터한테 "이거 해, 그다음 저거 해" 하고 적어주는 게 프로그래밍.
2
인터넷 vs 웹
인터넷 = 전 세계 컴퓨터를 잇는 도로망. 웹 = 그 도로로 문서(소포)를 주소(URL)에 배달하는 우편 시스템.
예문: "네이버 접속" = 내 컴퓨터가 인터넷(도로)을 통해 네이버 서버에 "그 페이지 줘" 요청 서버가 HTML 배달.

레벨 1 — 웹의 두 기둥 입문

3
프론트엔드 — 사용자 눈에 보이는 화면 (HTML·CSS·JavaScript)
4
백엔드 — 화면 뒤에서 데이터 처리하는 엔진 (서버·DB·API)
쉽게: 식당으로 치면 프론트엔드=홀(손님이 보는 곳), 백엔드=주방(요리·재료 관리). 둘이 맞물려 서비스가 굴러간다.

레벨 2 — 연결과 저장 초중급

5
API / REST — 프론트와 백엔드가 데이터 주고받는 약속(메뉴판). CRUD로 동작.
6
데이터베이스(DB) — 데이터를 구조적으로 저장(집=DB, 방=테이블). SQL로 다룸.

핵심 개념 깊게 — API · CRUD · 배포 초중급

API란? — 식당 메뉴판·웨이터

API(Application Programming Interface) = 프로그램끼리 데이터를 주고받는 약속된 창구. 손님(프론트)이 주방(백엔드) 내부를 몰라도 메뉴판(API)으로 주문하면 웨이터가 받아 요리를 가져온다. 주방이 어떻게 요리하는지 손님은 알 필요 없음 — 그게 핵심.

쉽게: 날씨앱이 기상청 데이터를 직접 캐오는 게 아니라, 기상청이 열어둔 API(창구)에 "오늘 서울 날씨 줘" 요청 → 정해진 형식으로 받아옴.

API 종류 — 통신 방식의 차이

종류특징쉬운 비유
REST (가장 흔함)HTTP 주소(URL)+메서드로 자원을 다룸. 단순·표준메뉴판: 정해진 메뉴를 번호로 주문
GraphQL필요한 데이터만 콕 집어 한 번에 요청. 과/부족 응답 방지뷔페 주문서: 원하는 것만 체크해 한 접시에
gRPC서버↔서버 초고속 통신. 이진(binary) 형식. MSA에서주방끼리 직통 전화: 빠르고 군더더기 없음
WebSocket한 번 연결해 양방향 실시간. 채팅·알림전화 통화: 끊지 않고 계속 주고받음
SOAP (구식)엄격한 XML 규격. 은행·기업 레거시정부 공문서: 형식 까다롭지만 엄격
입문은 REST부터. 95%의 시작점. 나머지는 필요할 때(실시간=WebSocket, 데이터 효율=GraphQL).
헷갈림 주의 — FastAPI는 API 종류가 아니다
예시정체
API 설계 방식REST · GraphQL · gRPC"어떻게 주고받을지" 약속·스타일 (설계도)
프레임워크FastAPI · Express · Spring그 API를 실제로 만드는 연장 (공구)
쉽게: REST = 집 설계도(어떤 방식으로 지을지). FastAPI = 그 설계도대로 짓는 공구(Python용). Express = 같은 일을 하는 다른 공구(JS용). 즉 "FastAPI로 REST API를 만든다"가 맞는 말. → 자세히는 FastAPI 탭.

REST = 주소 + HTTP 메서드

REST는 "자원을 URL로 표현하고, 행동은 HTTP 메서드로". 같은 주소 /todos라도 메서드가 다르면 다른 동작.

GET    /todos      → 목록 조회 (읽기)
POST   /todos      → 새 할 일 추가 (생성)
PATCH  /todos/3    → 3번 수정
DELETE /todos/3    → 3번 삭제
쉽게: /todos는 "할 일 게시판" 주소. GET=구경, POST=글쓰기, PATCH=수정, DELETE=지우기. 주소는 명사, 동작은 메서드.

CRUD란? — 모든 앱의 4가지 기본 동작

CRUD = Create(생성)·Read(조회)·Update(수정)·Delete(삭제). 거의 모든 서비스는 이 4개의 조합이다. 글쓰기=Create, 피드보기=Read, 수정=Update, 삭제=Delete.

CRUD뜻HTTP(REST)SQL(DB)예시
Create생성POSTINSERT게시글 작성
Read조회GETSELECT피드·목록 보기
Update수정PUT/PATCHUPDATE프로필 변경
Delete삭제DELETEDELETE댓글 지우기
한 줄: 프론트의 버튼 클릭 → REST 메서드 → SQL 쿼리 → DB. CRUD는 이 한 줄을 관통하는 공통 언어. 인스타·쿠팡·은행앱 전부 결국 CRUD.

클라우드 vs Vercel 배포 — 뭐가 다른가

둘 다 "내 서비스를 인터넷에 올리는 것"이지만 직접 관리하는 정도가 다르다.

클라우드(AWS·VPS 직접)Vercel 같은 플랫폼(PaaS)
비유빈 땅 빌려 집 직접 지음풀옵션 오피스텔 입주
해야 할 일서버 OS·웹서버·SSL·배포 다 직접 설정git push 한 번 → 자동 배포·HTTPS·CDN
자유도최고 (뭐든 가능)제한적 (정해진 틀 안)
난이도높음·관리 부담매우 쉬움·초보 적합
적합대규모·특수요구·비용통제프론트·Next.js·빠른 출시
쉽게: 클라우드(AWS) = 빈 땅 + 공구 받고 집 직접 짓기(자유롭지만 다 내 몫). Vercel = 가구 다 갖춰진 집에 짐만 옮기기(git push면 끝). Vercel도 사실 내부적으로 AWS 클라우드 위에서 돈다 — 어려운 부분을 대신 처리해주는 포장지.
정리: "클라우드"는 빌리는 인프라(땅·컴퓨터), "Vercel"은 그 위에서 배포를 자동화한 서비스. 반대 개념이 아니라 층이 다름. 입문은 Vercel로 시작 → 필요해지면 클라우드 직접. ( "서비스 만들기" 탭 STEP 7~8)
