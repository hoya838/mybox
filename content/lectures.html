강의 — 한 서비스가 도는 과정을 따라가며 배우기

"온라인 책방"을 하나 만든다고 상상하자. 손님이 책을 주문하는 순간부터 끝까지 따라가면, 웹개발 전체가 자연스럽게 이어진다. 외울 필요 없이 흐름으로.

이 강의의 약속: 흩어진 용어 암기 X. 책방 주문 시나리오 하나를 처음부터 끝까지 따라간다. 각 장면에서 필요한 기술이 "왜 여기서 등장하는지" 자연스럽게 만난다.

장면 0. 우리가 만들 것

온라인 책방. 손님이 책 목록을 보고 → 주문하고 → 주문내역을 확인한다. 딱 이 세 가지. 이 한 바퀴 안에 프론트·백엔드·DB·API·배포가 전부 들어있다. 하나씩 따라가 보자.

장면 1. 손님이 화면을 본다 — 프론트엔드

손님이 우리 사이트에 들어오면 화면이 보인다. 이 화면을 만드는 게 프론트엔드, 세 가지로 이뤄진다.

먼저 HTML로 뼈대를 세운다. 무엇이 화면에 있는지(제목·목록·버튼).

<h1>우리 책방</h1>
<ul id="book-list"></ul>        <!-- 책 목록이 들어갈 자리 -->
<button id="buy">주문하기</button>
쉽게: 집의 골조·벽. 여는 <h1>·닫는 </h1> 짝으로 "여긴 제목"을 표시.

뼈대만 있으면 밋밋하다. CSS로 색·크기·배치를 입힌다.

h1 { color: #6ea8fe; font-size: 32px; }
button { background: black; color: white; padding: 10px 20px; }
쉽게: HTML이 골조면 CSS는 페인트·인테리어. 화면(그릇)이 작아지면 모양이 바뀌게 하는 게 반응형 디자인 — 폰에선 메뉴를 햄버거(☰)로 접는 식.

그런데 화면은 가만히 있으면 안 된다. 버튼을 누르면 움직여야 한다. 그게 JavaScript.

const btn = document.querySelector('#buy');
btn.addEventListener('click', () => {
  // 주문하기를 누르면... 여기서 무슨 일이 일어날까? (장면 2로)
});
쉽게: 집의 전기·움직이는 장치. "버튼 누르면(이벤트) → 이렇게 해라". 그리고 모두가 쓰게 하려면(접근성) 이미지에 alt 설명을, 헤매지 않게(UX) 버튼은 버튼처럼 보이게.

장면 2. "주문하기"를 누르면? — 요청(Request)

손님이 버튼을 눌렀다. 그런데 책 주문 정보는 우리 서버 컴퓨터로 보내야 저장된다. 내 브라우저(클라이언트)가 서버에 메시지를 보낸다. 이걸 요청(request)이라 한다.

fetch('/orders', {
  method: 'POST',                 // 타입: 새로 만든다
  body: JSON.stringify({ bookId: 3, qty: 1 })  // 무슨 책 몇 권
});
요청 안에 든 것: 타입(POST=생성)·목적지(도메인)·경로(/orders)·내용(주문 정보).
쉽게: 식당에서 점원에게 주문서를 건네는 것. "이 책, 한 권요." 어디로 가는지(도메인)·뭘 하려는지(타입+경로)가 적혀 있다.

여기서 경로 이름 짓는 약속이 REST다. 경로는 명사(/orders), 동작은 타입(메서드)으로.

POST   /orders     주문 생성
GET    /orders     주문 목록 조회
DELETE /orders/3   3번 주문 취소
쉽게: 메뉴판 약속. 명사(주문)+동사(만들기/보기/지우기)로 나눈다. REST 말고 GraphQL·RPC 약속도 있지만, 입문은 REST.

장면 3. 서버가 메시지를 받는다 — 백엔드

요청이 서버에 도착했다. 그런데 컴퓨터는 원래 인터넷 메시지를 못 받는다. 받을 수 있게 프로그래밍해야 한다. 그래서 백엔드 언어가 필요하다(Python·JavaScript·Java…). 입문은 Python이나 Node.js.

맨바닥부터 짜면 코드가 너무 많다. 프레임워크(서버 조립 키트)를 쓴다.

# FastAPI (Python) — "POST /orders 오면 이렇게 처리해라"
@app.post("/orders")
def create_order(order):
    # 1) 주문을 저장하고  2) 결과를 돌려준다 (장면 4로)
    ...
쉽게: 골격은 프레임워크가 깔아주고, 나는 "주문 오면 뭘 할지"만 채운다. 그리고 남이 만든 기능(라이브러리)은 패키지 매니저로 설치 — pip install fastapi / npm install express. "바퀴를 다시 발명하지 마라."

장면 4. 주문을 저장한다 — 데이터베이스

서버가 주문을 받았으니 어딘가 저장해야 한다. 안 그러면 껐다 켜면 사라진다(그게 변수). 영구 보관은 데이터베이스(DB). 대표는 PostgreSQL·MySQL.

-- 주문 한 건 저장 (Create)
INSERT INTO orders (book_id, qty) VALUES (3, 1);
-- 내 주문 다 보기 (Read)
SELECT * FROM orders WHERE user_id = 12;
쉽게: DB=엑셀 창고, 테이블=종류별 시트. 다루는 4동작이 CRUD(생성·조회·수정·삭제) — 장면 2의 REST 메서드(POST·GET·PATCH·DELETE)와 정확히 짝.
흐름 점검: 버튼 클릭 → POST /orders(요청) → FastAPI가 받음 → INSERT로 DB 저장. 절반 왔다. 이제 손님에게 결과를 돌려줄 차례.

장면 5. 결과가 화면으로 돌아온다 — 응답(Response)

서버가 저장을 마치고 손님에게 "주문 완료!"를 보낸다. 이게 응답(response). 요청↔응답 한 바퀴를 요청-응답 사이클이라 하고, 이게 모든 웹서비스가 도는 방식이다.

// 프론트: 응답을 받아 화면 갱신
fetch('/orders').then(r => r.json()).then(orders => {
  // 받은 주문 목록을 화면(HTML)에 그린다
});
쉽게: 점원이 주방(DB)에 주문 넣고 → "나왔습니다" 하고 음식을 가져다주는 것. 프론트는 그걸 받아 화면에 다시 그린다. 이 API 연동이 프론트·백을 잇는 다리.

장면 6. "근데 너 누구야?" — 인증·인가

주문하려면 로그인이 필요하다. 인증=너 누구냐(로그인), 인가=뭘 할 수 있냐(권한).

예문: 로그인(인증) → 토큰 발급 → 내 주문만 조회 가능(인가). 남의 주문 삭제 시도 → 거부(403).
쉽게: 인증=출입증, 인가=갈 수 있는 층. 비밀번호를 직접 관리하면 위험하니 보통 구글·카카오 로그인(소셜 로그인)으로 맡긴다 — "비밀번호를 내가 안 갖는다"가 핵심.

장면 7. 세상에 공개한다 — 배포·클라우드

지금까진 내 컴퓨터(로컬)에서만 돈다. 친구도 쓰게 하려면 인터넷에 올려야 한다 = 배포. 서버 컴퓨터는 직접 사지 않고 빌린다(클라우드).

단계뜻쉽게
IaaS가상 컴퓨터를 빌림(AWS·GCP)빈 땅 임대
PaaS코드만 올리면 자동 배포(Vercel·Railway)풀옵션 입주
SaaS완성 기능을 API로 빌림(이메일·결제)완제품 구독
쉽게: 손님 몰리면(명절) 서버 여러 대로 늘리고 로드밸런서가 요청을 나눠준다(작업반장). 입문은 PaaS(Vercel+Supabase)로 git push 한 번이면 끝.

장면 8. 더 빠르고 튼튼하게 — 운영 기술

서비스가 돌기 시작하면 다듬는다. 이 단계 기술들은 흐름의 곳곳을 보강한다.

1
테스트·디버깅 — 의도대로 도는지 확인하고 버그 잡기. console.log로 값 추적. 코딩 시간의 80%.
2
브라우저 개발자도구(F12) — 화면의 X-레이. HTML·네트워크 요청·콘솔 오류 다 보임.
3
성능 최적화 — 이미지 압축·지연로딩·캐시(Redis)로 빠르게. "3초 안 뜨면 절반 이탈."
4
Git — 코드 세이브 포인트·협업. 처음부터 끝까지 매일 commit·push.

장면 9. 규모가 커지면 — 심화 선택지

책방이 잘 돼서 코드가 커지면 다음을 만난다. 필요해질 때 배우면 된다.

TypeScript
JS에 안전벨트. "이건 숫자" 미리 정해 오타·타입 오류를 코딩 중 차단. 큰 프로젝트 필수.
React / Next.js
화면을 레고 블록(컴포넌트)으로. 버튼·카드를 부품화해 재사용. Next는 SSR로 SEO·속도까지.
MSA
백엔드를 기능 단위(주문·결제·이메일)로 쪼갬. 각각 다른 언어·DB 가능.
SEO
검색에 잘 걸리게. 제목 태그·설명·빠른 속도가 점수.
보조 DB
이미지=S3+CDN, 검색=Elasticsearch, 캐시=Redis, 분석=Snowflake. 주 DB는 운영에 집중.
한 바퀴 정리: 화면(프론트) → 클릭하면 요청(REST) → 서버가 받고(백엔드·프레임워크) → DB 저장(CRUD) → 응답으로 화면 갱신 → 로그인(인증) → 배포(클라우드). 이 한 흐름이 모든 웹서비스의 뼈대. 책방을 한 번 끝까지 만들면 어떤 서비스든 이 틀의 확장이다.
다음: 각 장면을 작은 프로젝트로 직접 만들어보기(게시판·Todo면 충분). 그다음 AI 도구(클로드·코덱스)로 속도를 올린다. — 입력만 말고 직접 만들어야 실력이 된다.
