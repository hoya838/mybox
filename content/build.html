서비스 만들기 — 할 일(Todo) 앱 풀스택

배운 언어를 합쳐 하나의 진짜 서비스로. 기획→DB→백→프론트→연결→배포.

전체 그림: 사용자가 브라우저(React)에서 할 일 추가 → API(FastAPI/Express)가 받아 → DB(PostgreSQL)에 저장 → 다시 화면에 표시. 이 한 바퀴가 모든 웹서비스의 기본.

STEP 1. 기획 — MVP 정하기 기초

기능 욕심 버리고 핵심 하나. Todo 앱이면: 할 일 추가·목록보기·완료체크. 끝.

왜: 카카오톡도 처음엔 "대화"만. 작게 시작해 굴러가게 먼저.

STEP 2. DB 설계 — PostgreSQL 기초

데이터를 어떻게 저장할지. 할 일 = 테이블 하나.

CREATE TABLE todos (
  id SERIAL PRIMARY KEY,
  title TEXT NOT NULL,
  done BOOLEAN DEFAULT false
);
이건: "할일" 방(테이블)에 id·제목·완료여부 칸. DB

STEP 3. 백엔드 API — FastAPI(또는 Express) 중급

프론트가 호출할 통로(CRUD). 같은 걸 두 언어로 — 하나 골라 시작.

# FastAPI (Python)
@app.get("/todos")          # 목록
def list_todos(): ...
@app.post("/todos")         # 추가
def add_todo(t: Todo): ...
@app.patch("/todos/{id}")   # 완료체크
def toggle(id: int): ...
// Express (Node) — 같은 API
app.get('/todos', ...);
app.post('/todos', ...);
app.patch('/todos/:id', ...);
이건: "할일 줘/추가해/체크해" 요청을 받아 DB에 반영. 백엔드

STEP 4. 프론트엔드 — React 중급

사용자가 보는 화면. API를 fetch로 호출해 표시.

function App() {
  const [todos, setTodos] = useState([]);
  useEffect(() => {
    fetch('/todos').then(r => r.json()).then(setTodos);
  }, []);
  return todos.map(t => <li key={t.id}>{t.title}</li>);
}
이건: 서버에서 할일 받아 목록 그림. 추가 버튼은 POST 호출. 프론트

STEP 5. 연결 — 프론트 ↔ 백 중급

React가 fetch로 API 호출 → API가 DB 조회 → 응답을 화면에. CORS 설정 필요.

데이터 한 바퀴:
React(추가 클릭) → POST /todos → FastAPI → INSERT → PostgreSQL
PostgreSQL → FastAPI → JSON → React(목록 갱신)

STEP 6. 배포 — 세상에 공개 심화

내 컴퓨터(localhost) → 인터넷. 도커로 포장 → 클라우드(VPS) 서버 → 도메인 연결.

1
백엔드+DB를 서버(클라우드)에 올림 (도커)
2
프론트는 정적 호스팅(Vercel 등)
3
도메인 사고 DNS 연결 → 누구나 접속

STEP 7. 어디에 올릴까 — 배포·서비스 추천 심화

웹서비스는 3개 층으로 나눠 각각 다른 곳에 올린다. 한 곳에 다 넣을 필요 없음 — 층마다 잘하는 플랫폼이 다르다.

3층 구조: ① 프론트(사용자 화면·정적/SSR) ② 백엔드(API 서버·항상 켜짐) ③ DB(데이터 저장). 플랫폼은 이 중 하나 또는 둘을 담당한다.

플랫폼별 정리

프론트(화면) — Vercel · Netlify · 카페24

Vercel 프론트 서버리스 — Next.js 만든 회사. git push 한 번에 배포·미리보기·HTTPS 자동·전세계 CDN. Next.js면 1순위. API Routes(서버리스 함수)도 같이 돌아가 가벼운 백엔드는 따로 안 필요.

한계: 항상 켜진 서버(웹소켓·장시간 작업·무거운 Python)는 부적합. 함수가 잠깐 떴다 꺼지는 구조. DB 없음 → Supabase 등 붙여야 함.

Netlify — Vercel과 거의 동급 대안. 정적·JAMstack 강함. Next.js 최적화는 Vercel이 우위.

백엔드+DB 한방 — Supabase

Supabase 백엔드 DB — "백엔드를 코드 없이". PostgreSQL + 회원가입/로그인(Auth) + 파일저장(Storage) + 실시간 + 자동 API를 한 번에. STEP 2~3(DB·API)을 직접 안 짜도 프론트에서 바로 호출.

이건: 테이블만 만들면 supabase.from('todos').select()처럼 프론트에서 DB 직접 호출. 인증도 함수 한 줄. 풀 Postgres라 확장(pgvector 등) 제한 없음.
Vercel과 관계: 경쟁 아님, 짝꿍. Vercel=프론트, Supabase=백엔드·DB. 2026년 Next.js 앱의 기본 스택. 각각 무료→유료 $20·$25 별도.

내 서버 직접 운영 — Railway · Render

Railway / Render 백엔드 인프라 — FastAPI·Express 같은 내가 짠 서버 코드를 컨테이너로 직접 올림(Heroku 후속). Postgres도 클릭 한 번 추가. Vercel이 못하는 "항상 켜진 서버"를 담당.

RailwayRender
과금분당 사용량 기반(쓴 만큼)월 고정·넉넉한 무료티어
성향속도·유연·변동 워크로드예산 예측·취미/소규모 안정
Supabase와 차이: Supabase는 "이미 만들어진 백엔드"를 빌려쓰는 것, Railway/Render는 "내 백엔드 코드"를 올리는 것. 커스텀 로직 많으면 후자.

한국 서비스 — 카페24

카페24 한국 호스팅 — 국내 전통 호스팅. 종류가 여럿:

상품용도
웹호스팅(뉴아우토반)정적 사이트·PHP — 가장 쉬움
Node.js 호스팅JS 서버 구동. git push 방식(Heroku 유사), FTP 아님
가상서버(VPS)리눅스 직접 구축 — 자유도 최고, 직접 관리
AI SPACE신규. 월 4,900원·한국서버·DB 내장·한국어 — Vercel/Railway 국내 대안
언제: 사용자가 한국 위주(속도), 국내 결제·사업자·고객센터가 필요, 쇼핑몰. 주의: Vercel/Supabase보다 배포가 번거롭다(git 터미널·SSL 설정). 편의성은 해외 플랫폼 우위.

전체 상관관계 한눈에

데이터 흐름과 담당 플랫폼:
브라우저 → [Vercel] 프론트(React/Next) → API 호출
  ↳ 가벼우면 [Vercel] 서버리스 함수에서 처리
  ↳ 백엔드 빌려쓰기 [Supabase] (Auth·DB·Storage 자동)
  ↳ 내 서버 직접 [Railway/Render] FastAPI·Express → [Postgres]
국내 통합 운영 → [카페24 가상서버 / AI SPACE]
한 줄 정리: Next.js 입문이면 Vercel + Supabase로 시작(공짜·코드 최소). 백엔드를 직접 깊게 짜려면 Railway/Render 추가. 한국 타깃·쇼핑몰이면 카페24. 셋은 경쟁이 아니라 역할 분담이다.
