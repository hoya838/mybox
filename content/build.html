서비스 만들기 — 할 일(Todo) 앱 풀스택

배운 언어를 합쳐 하나의 진짜 서비스로. 기획→DB→백→프론트→연결→배포.

다 배우고 오지 마라: 학습법(거꾸로 배우기)대로 여기부터 부딪히는 게 맞다. 막히는 지점만 커리큘럼으로 내려가 "왜?"를 물어라.
전체 그림: 사용자가 브라우저(React)에서 할 일 추가 → API(FastAPI/Express)가 받아 → DB(PostgreSQL)에 저장 → 다시 화면에 표시. 이 한 바퀴가 모든 웹서비스의 기본.

STEP 1. 기획 — MVP 정하기 기초

기능 욕심 버리고 핵심 하나. Todo 앱이면: 할 일 추가·목록보기·완료체크. 끝.

왜: 카카오톡도 처음엔 "대화"만. 작게 시작해 굴러가게 먼저.

코드 짜기 전에 — 기획 문서부터

바로 코딩 X. 먼저 "무엇을 왜 어떻게 만들지"를 글로 적는다. 이 문서들이 곧 AI(Claude Code·Cursor)에게 주는 설계도이자, 나중에 길 잃지 않는 지도가 된다. 프로젝트 폴더에 docs/ 만들어 마크다운(.md)으로 저장.

PRD 예시 (prd.md)

# Todo 앱 PRD

## 왜 (문제)
할 일을 메모앱에 적으면 흩어지고 완료관리가 안 됨.

## 누구 (사용자)
혼자 일정 관리하는 개인.

## 핵심 가치
"추가-확인-완료" 3초 안에. 군더더기 없음.

## MVP 범위 (P0 = 이번에 꼭)
- [ ] 할 일 추가
- [ ] 목록 보기
- [ ] 완료 체크
## 다음 (P1 = 나중)
- 마감일, 카테고리, 검색
이건: "왜·누구·핵심·이번엔 뭐까지" 한 장. 기능 추가 욕심날 때 P1로 미루는 기준이 됨.

기능명세서 예시 (spec.md)

# 기능: 완료 체크

- 입력: 할 일 항목의 체크박스 클릭
- 처리: done 값을 true↔false 토글, DB 저장
- 결과: 텍스트에 취소선, 목록 맨 아래로 이동
- 예외: 저장 실패 시 체크 원위치 + "다시 시도" 토스트
이건: 기능 하나의 입력→처리→결과→예외. 이대로 AI에게 주면 거의 그대로 코드가 나온다. 모호함이 적을수록 버그가 준다.
AI 코딩 시대 팁: 이 문서들을 레포의 docs/·CLAUDE.md에 두면 Claude Code·Cursor가 읽고 일관된 코드를 만든다. 좋은 기획서 = 좋은 프롬프트. (vault "서비스 개발 전체과정"·하네스 설계 참고)

STEP 2. DB 설계 — PostgreSQL 기초

데이터를 어떻게 저장할지. 할 일 = 테이블 하나.

CREATE TABLE todos (
  id SERIAL PRIMARY KEY,
  title TEXT NOT NULL,
  done BOOLEAN DEFAULT false
);
이건: "할일" 방(테이블)에 id·제목·완료여부 칸. DB

STEP 3. 백엔드 API — FastAPI(또는 Express) 중급

프론트가 호출할 통로(CRUD). 같은 걸 두 언어로 — 하나 골라 시작.

# FastAPI (Python)
@app.get("/todos")          # 목록
def list_todos(): ...
@app.post("/todos")         # 추가
def add_todo(t: Todo): ...
@app.patch("/todos/{id}")   # 완료체크
def toggle(id: int): ...
// Express (Node) — 같은 API
app.get('/todos', ...);
app.post('/todos', ...);
app.patch('/todos/:id', ...);
이건: "할일 줘/추가해/체크해" 요청을 받아 DB에 반영. 백엔드

STEP 4. 프론트엔드 — React 중급

사용자가 보는 화면. API를 fetch로 호출해 표시.

function App() {
  const [todos, setTodos] = useState([]);
  useEffect(() => {
    fetch('/todos').then(r => r.json()).then(setTodos);
  }, []);
  return todos.map(t => <li key={t.id}>{t.title}</li>);
}
이건: 서버에서 할일 받아 목록 그림. 추가 버튼은 POST 호출. 프론트

STEP 5. 연결 — 프론트 ↔ 백 중급

React가 fetch로 API 호출 → API가 DB 조회 → 응답을 화면에. CORS 설정 필요.

데이터 한 바퀴:
React(추가 클릭) → POST /todos → FastAPI → INSERT → PostgreSQL
PostgreSQL → FastAPI → JSON → React(목록 갱신)

STEP 6. 배포 — 세상에 공개 심화

내 컴퓨터(localhost) → 인터넷. 도커로 포장 → 클라우드(VPS) 서버 → 도메인 연결.

1
백엔드+DB를 서버(클라우드)에 올림 (도커)
2
프론트는 정적 호스팅(Vercel 등)
3
도메인 사고 DNS 연결 → 누구나 접속

STEP 7. 어디에 올릴까 — 배포·서비스 추천 심화

웹서비스는 3개 층으로 나눠 각각 다른 곳에 올린다. 한 곳에 다 넣을 필요 없음 — 층마다 잘하는 플랫폼이 다르다.

3층 구조: ① 프론트(사용자 화면·정적/SSR) ② 백엔드(API 서버·항상 켜짐) ③ DB(데이터 저장). 플랫폼은 이 중 하나 또는 둘을 담당한다.

먼저: 클라우드 서버가 뭔데?

클라우드 서버 = 데이터센터에 있는 남의 컴퓨터를 인터넷으로 빌려 쓰는 것. 내 노트북이 아니라 24시간 켜진 원격 컴퓨터다. 사서 관리하지 않고 월세처럼 빌리면 고장·전기·확장·보안을 빌려주는 회사가 처리한다.

비유: 식당을 차릴 때 건물을 직접 짓는 대신 임대하는 것. 내 서비스(코드)를 클라우드에 올리면 전세계 누구나 24시간 접속 가능. Vercel·Supabase·Railway도 전부 내부적으로 이 클라우드(AWS 등) 위에서 돈다.

SQL·PostgreSQL은 Supabase 안에 이미 들어있다

따로 DB 서버를 두지 않는다. Supabase = 클라우드에서 PostgreSQL을 대신 돌려주는 서비스 + Auth·Storage·자동 API를 얹은 것. 가입하면 Postgres 1개가 자동 생성된다.

즉: STEP 2의 CREATE TABLE todos ... SQL을 Supabase 대시보드의 SQL Editor에 붙여넣고 실행하면 끝. DB·데이터·SQL이 전부 Supabase 한 곳에 있고 백엔드와 한 덩어리. 별도 Postgres 호스팅을 살 필요 없음.

플랫폼별 정리

프론트(화면) — Vercel · Netlify · 카페24

Vercel 프론트 서버리스 — Next.js 만든 회사. git push 한 번에 배포·미리보기·HTTPS 자동·전세계 CDN. Next.js면 1순위. API Routes(서버리스 함수)도 같이 돌아가 가벼운 백엔드는 따로 안 필요.

한계: 항상 켜진 서버(웹소켓·장시간 작업·무거운 Python)는 부적합. 함수가 잠깐 떴다 꺼지는 구조. DB 없음 → Supabase 등 붙여야 함.

Netlify — Vercel과 거의 동급 대안. 정적·JAMstack 강함. Next.js 최적화는 Vercel이 우위.

백엔드+DB 한방 — Supabase

Supabase 백엔드 DB — "백엔드를 코드 없이". PostgreSQL + 회원가입/로그인(Auth) + 파일저장(Storage) + 실시간 + 자동 API를 한 번에. STEP 2~3(DB·API)을 직접 안 짜도 프론트에서 바로 호출.

이건: 테이블만 만들면 supabase.from('todos').select()처럼 프론트에서 DB 직접 호출. 인증도 함수 한 줄. 풀 Postgres라 확장(pgvector 등) 제한 없음.
Vercel과 관계: 경쟁 아님, 짝꿍. Vercel=프론트, Supabase=백엔드·DB. 2026년 Next.js 앱의 기본 스택. 각각 무료→유료 $20·$25 별도.

내 서버 직접 운영 — Railway · Render

Railway / Render 백엔드 인프라 — FastAPI·Express 같은 내가 짠 서버 코드를 컨테이너로 직접 올림(Heroku 후속). Postgres도 클릭 한 번 추가. Vercel이 못하는 "항상 켜진 서버"를 담당.

RailwayRender
과금분당 사용량 기반(쓴 만큼)월 고정·넉넉한 무료티어
성향속도·유연·변동 워크로드예산 예측·취미/소규모 안정
Supabase와 차이: Supabase는 "이미 만들어진 백엔드"를 빌려쓰는 것, Railway/Render는 "내 백엔드 코드"를 올리는 것. 커스텀 로직 많으면 후자.

STEP 7-2. 회원가입 — 왜 구글·카카오 로그인을 쓰나 중급

자체 이메일+비밀번호 가입을 직접 만들지 않고 구글·카카오 외부 계정 연동(소셜 로그인 = OAuth)으로 가입만 붙이는 이유. 핵심 한 줄: "비밀번호를 내가 안 갖는다."

이유설명
① 보안 책임 떠넘김 (가장 큼)비번을 내 DB에 저장 안 함 → 유출 사고 자체가 불가능(없는 걸 못 턴다). 해싱·암호화·무차별대입 방어를 구글·카카오가 대신
② 개발 비용 ↓이메일 인증·비번 찾기·재설정·2단계 인증 전부 안 만들어도 됨 (직접 만들면 일주일+)
③ 가입 전환율 ↑폼 작성+이메일 인증(이탈 多) vs 버튼 한 번. 한국은 카카오 없으면 이탈
④ 검증된 계정이미 인증된 이메일·실사용자 → 봇·가짜계정 ↓
OAuth 흐름 (비번이 안 오감):
사용자 "구글로 로그인" 클릭 → [구글]이 인증(비번은 구글만 봄) → 내 서버에 토큰 전달 → 그 토큰으로 사용자 식별·로그인
쉽게: 클럽 입구에서 신분증 직접 안 받고 "정부(구글)가 보증하는 도장"만 확인하는 것. 내 가게는 주민번호(비번)를 보관할 필요가 없어 털릴 일도 없다. Supabase Auth면 소셜 버튼 한 줄
트레이드오프(주의): ① 외부 의존 — 구글 다운 시 가입 불가 ② 그 계정 없는 사용자는 못 씀 → 보통 여러 개 제공(구글+카카오+이메일) ③ 제공자 정책 변경 리스크. 실무 정석: 소셜 로그인 우선 + 이메일 가입도 보조로. 비번 직접 관리는 가능한 피한다.

STEP 8. 헷갈림 정리 — API 키·데이터는 어디에, 서버는 몇 개? 심화

API 키는 어디에 두나 (2종류)

키 종류예시저장 위치
① Supabase 접속 키anon key, service_role keySupabase가 발급 → 호스팅 환경변수에 넣어 사용
② 외부 서비스 키OpenAI, 토스/Stripe 결제, 카카오 로그인Supabase 아님 — 호스팅(Vercel/Railway) 환경변수
