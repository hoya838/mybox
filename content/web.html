웹개발 — 프론트·백엔드·클라우드·API

필수 용어 + 예문 + 쉬운 해석 + 언어 정리. 강의 탭이 "흐름"이면, 여기는 그 흐름에 나온 용어를 찾아보는 "사전".

큰 그림: 이 용어들이 5층 지식 구조의 어디에 놓이는지는 AI 학습 5요소 지도에서. 프론트·백·인프라가 "클릭 하나가 지나가는 5정거장"으로 정리돼 있다.
한 흐름으로 먼저: 손님이 화면(①프론트)을 보고 버튼을 누르면 → 서버에 요청(③API)이 가고 → 백엔드(②)가 받아 → DB(④)에 저장하고 → 응답을 화면에 돌려준다. 이걸 인터넷에 올리면(⑤클라우드) 끝. 아래 ①~⑥은 이 한 바퀴의 부분들이다.

① 프론트엔드 (Front-end)

흐름상 여기 — 손님이 처음 만나는 화면. 사용자가 직접 보고 클릭하는 화면. 브라우저에서 돌아감.

기술역할쉽게
HTML화면 뼈대·내용집의 골조·벽
CSS꾸미기(색·폰트·배치)페인트·인테리어
JavaScript동작(버튼·이벤트)전기·움직이는 장치
React·Vue·Next화면을 효율적으로 만드는 도구조립식 가구 키트
예문: 쇼핑몰 "장바구니 담기" 버튼을 눌렀을 때 숫자가 +1 되는 것 = JavaScript가 처리한 이벤트.
쉽게: React는 사실 "프레임워크"처럼 보여도 갖다 붙이는 라이브러리에 가깝다. 왜 생겼는지 알면 외울 필요 없음.

SPA / MPA / CSR / SSR — 렌더링 용어 정리

프론트 공부하다 자주 나오는 4개 용어. 두 개의 독립된 축이다.

SPAMPA
뜻Single Page App — HTML 파일 1개Multi Page App — HTML 파일 여러 개
페이지 이동JS가 DOM 갈아끼움, 새로고침 없음새 HTML 요청·로드, 깜빡임 있음
비유방 하나, 가구만 바꿈방 바꿀 때마다 다른 방으로 이동
예시Gmail, 트위터쇼핑몰 상품페이지, 블로그
CSRSSR
뜻Client-Side Rendering — 브라우저가 HTML 생성Server-Side Rendering — 서버가 HTML 생성
흐름서버 → 빈 HTML+JS → 브라우저가 화면 그림서버 → 완성된 HTML → 바로 표시
첫 화면느림 (JS 실행 대기)빠름
SEO약함 (크롤러가 빈 HTML만 봄)강함
비유재료만 주고 손님이 직접 요리주방에서 완성해서 냄
SPA/MPA = 건물 구조(방 몇 개?), CSR/SSR = 요리 방식(누가 만드나?)
독립된 두 축 — 조합 가능, 상하위 없음.
조합예시특징
SPA + CSRReact (CRA)첫 로딩 느림, SEO 약함
SPA + SSRNext.js첫 HTML은 서버, 이후 JS가 SPA처럼 동작
MPA + SSRPHP · Django전통 웹, 페이지마다 서버 렌더
Next.js가 왜 쓰이나: CSR 단점 3개(첫 로딩 느림·SEO 약함·프론트+백 서버 두 개 관리)를 SSR+SPA 조합으로 동시에 해결. 첫 접속만 서버가 완성 HTML 주고, 이후는 JS가 SPA처럼 처리 — 이걸 하이드레이션(Hydration)이라 함.

② 백엔드 (Back-end)

흐름상 여기 — 요청을 받아 처리하는 뒷단. 화면 뒤에서 데이터를 처리·저장하는 엔진. 서버에서 돌아감.

탄생 이유 = 동적 생성. 옛날엔 서버가 HTML 파일을 그냥 꺼내줬는데, 방명록·검색은 "누가 뭘 남길지" 미리 못 만듦 요청 올 때 그 자리에서 만들어야 했다(1993 CGIPHP…).
구성역할
서버요청 받고 응답하는 컴퓨터(24시간 켜짐)
WAS웹 애플리케이션 서버 — 로직 실행
언어/프레임워크Node.js·Python(Django)·Java(Spring)·PHP
예문: 로그인 = 프론트가 보낸 아이디/비번을 백엔드가 받아 DB의 회원 정보와 대조 맞으면 "통과" 응답.

웹서버 vs WAS — 뭐가 다른가

둘 다 "서버"지만 역할이 다르다. 웹서버는 앞단 문지기, WAS는 뒷단 주방.

웹서버 (Web Server)WAS (Web Application Server)
역할요청 받기 + 정적 파일(HTML·CSS·이미지) 그대로 전달동적 처리 — 로그인·DB조회·계산 등 로직 실행
처리이미 만들어진 파일을 꺼내 줌 (연산 거의 없음)요청마다 그 자리에서 결과를 만들어 냄
예시 제품Nginx · ApacheTomcat · Node.js · (Spring·Django 구동)
비유식당 카운터·홀 — 주문 받고 완성품 서빙, 정해진 건 바로 줌주방 — 주문대로 요리(연산)
요청 흐름:
브라우저 → [웹서버] 정적이면(이미지·HTML) 바로 응답
  ↳ 동적이면(로그인·검색) → [WAS] 로직 실행 → [DB] 조회 → 결과를 웹서버 거쳐 응답
왜 나눠? 웹서버가 앞에서 정적 요청·트래픽을 빠르게 쳐내고, 무거운 연산만 WAS로 넘겨 부하 분산·보안. 작은 프로젝트는 한 덩어리(Node 하나)로도 가능 — 커지면 분리.

③ API / REST

흐름상 여기 — 프론트↔백엔드가 요청·응답을 주고받는 통로. 데이터 주고받는 약속(메뉴판). "이렇게 요청하면 이렇게 응답한다."

REST = CRUD (가장 널리 쓰는 형식)
동작뜻HTTP 메서드예
Create생성POST새 글 등록
Read조회GET글 목록 보기
Update수정PUT/PATCH글 고치기
Delete삭제DELETE글 지우기
쉽게: API = 식당 점원. 손님(프론트)이 "김치찌개요"(요청) 점원이 주방(DB)에서 받아 "여기요"(응답). 손님이 주방 직접 안 들어감.
예문: GET /books/3 = "3번 책 정보 줘" 응답 200 OK + 책 데이터(JSON).

④ 데이터베이스 (DB)

흐름상 여기 — 백엔드가 데이터를 영구 저장하는 곳. 데이터를 구조적으로 저장·관리한다. DB=집, 테이블=방, 행/열=엑셀 시트.

용어쉽게
DB큰 창고(집)
테이블종류별 방(회원방·상품방)
SQLDB에 명령하는 언어("회원 다 줘")
스토리지파일(이미지·동영상) 저장 — DB와 별개(S3 등)

⑤ 클라우드 / 배포

흐름상 마지막 — 다 만든 서비스를 세상에 공개. 내 컴퓨터(로컬)에서만 돌던 걸 인터넷에 올려 남도 쓰게 한다.

용어뜻비유
서버요청에 응답하는 컴퓨터24시간 켜진 가게
IP인터넷상 컴퓨터 주소(숫자)집 주소
도메인IP를 사람이 읽는 이름(naver.com)가게 간판
DNS도메인IP 변환주소록
클라우드서버를 빌려 쓰는 서비스(AWS 등)전세 대신 월세
도커실행 환경 통째 포장이사 박스
클라우드 3계층 — 어디까지 빌리나
계층빌리는 것비유
IaaS빈 가상 컴퓨터(서버)만 빌림 — OS·서버설정 내가 다 함 (AWS EC2·GCP)빈 땅 임대 — 집은 내가 지음
PaaS코드만 올리면 서버·배포 자동 (Vercel·Railway·Heroku)풀옵션 오피스텔 입주
SaaS완성된 기능을 API로 빌려 씀 (이메일=Twilio·결제=Stripe·채널톡)완제품 구독
IaaS 쉽게: Infrastructure as a Service = "인프라(컴퓨터)를 서비스로 빌린다." AWS가 가진 거대한 서버 안의 가상 컴퓨터 한 대를 월세로 빌리는 것. 빈 컴퓨터라 OS 깔고 서버 설정도 내가 함(자유도 최고·관리 부담 큼). 트래픽 몰리면 여러 대 빌려 로드밸런서로 분배, 한가하면 끔.
예문: localhost:3000 = 내 컴퓨터에서만 열림. 친구도 보게 하려면 클라우드 서버에 배포 + 도메인 연결.

언어주 용도한 줄
