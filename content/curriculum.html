JavaScript프론트 + 백(Node)웹 공용어. 브라우저·서버 둘 다

React프론트 (라이브러리)화면을 부품으로 조립
Python백엔드서버·AI·데이터
FastAPI백엔드 (Python 웹 프레임워크)Python으로 API 서버 빠르게
Node.js백엔드 (JS 런타임)JavaScript를 서버에서 실행

쉽게: FastAPI·Express = "서버 만드는 틀"(백). PostgreSQL = "데이터 창고"(백). React = "화면 조립 키트"(프론트). 프레임워크는 언어 위에 얹는 골격이다.
내 백엔드 2갈래: Python → FastAPI로 서버 / JavaScript → Node.js + Express로 서버. 둘 다 PostgreSQL에 연결. 같은 일을 두 언어로 — 비교하며 배우면 개념이 또렷해짐.

1. 기술 전체 지도 — 화면·서버·데이터로 나누면

이 기술들은 각자 담당 구역이 다르다. 구역부터 잡으면 관계가 보인다.

🖥 화면 (프론트엔드)
HTML — 뼈대
CSS — 스타일
JavaScript — 동작
TypeScript — JS + 타입
React — 화면 조립 도구
Next.js — React + 서버렌더
⚙️ 서버 (백엔드)
Node.js — JS 서버 실행환경
Express — Node API 프레임워크
Python — 서버 언어
FastAPI — Python API 프레임워크
🗄 데이터 (DB)
PostgreSQL — 데이터 저장소
SQL — DB 조작 언어

프론트엔드 안에서 층층이 쌓이는 구조

HTML·CSS·JS는 기본 재료, 그 위에 도구들이 올라간다.

① 기본 재료
  HTML — 화면 뼈대 ("여기 제목, 여기 버튼")
  CSS — 색·크기·배치 꾸미기
  JavaScript — 버튼 클릭, 데이터 불러오기 등 동작

② JS에 안전장치 추가
  TypeScript — JS 그대로인데 타입 오류를 미리 잡아줌. "JS의 업그레이드판"
    → JS 코드 = TS에서 그대로 동작. TS ⊃ JS

③ JS로 화면을 더 쉽게 만드는 도구
  React — 화면을 작은 부품(컴포넌트)으로 나눠 조립. 상태(state) 바뀌면 자동 갱신
    → JS(또는 TS) 위에 올라가는 라이브러리

④ React에 서버 기능 추가
  Next.js — React로 만든 화면을 서버에서 미리 렌더링(SSR). 라우팅·API도 포함
    → React 위에 올라가는 프레임워크. "React가 못 하는 것까지"

백엔드 — JS가 서버로 가는 방법

문제: JS는 원래 브라우저 전용 언어
  → 서버에서 실행하려면 실행환경이 필요

Node.js — "JS를 서버에서 돌려주는 엔진(런타임)". 언어가 아님
  → Node 덕분에 JS 하나로 프론트+백 모두 가능해짐

Express — Node.js 위에 올라가는 웹 프레임워크
  → "이 URL로 요청 오면 이렇게 응답해라" 라우팅·미들웨어 제공
질문답
TypeScript랑 JavaScript 뭐가 달라?TS = JS + 타입 체크. JS 코드 그대로 TS에서 동작. 오타·타입 오류를 실행 전에 잡음
Node.js가 언어야?아님. JS를 서버에서 실행하는 환경(런타임). 언어는 그대로 JavaScript
React랑 Next.js 뭐가 달라?React = 화면만. Next = React + SSR + 라우팅 + API. React 위에 올라감
React가 프레임워크야?아님. 라이브러리. UI만 담당. 라우팅·서버는 React 밖(Next.js가 해결)
Express랑 FastAPI 차이?언어만 다름. Express = Node.js(JS)로 서버. FastAPI = Python으로 서버. 역할 같음

예시 — "게시글 목록 불러오기"

사용자가 게시판 페이지를 열었을 때 일어나는 일을 단계별로 본다.

1
프론트 (React) — 서버에 요청
컴포넌트가 화면에 뜨는 순간 서버로 HTTP GET 요청을 보낸다 
// React 컴포넌트
useEffect(() => {
  fetch('http://localhost:4000/posts')   // 서버 주소로 GET 요청
    .then(res => res.json())
    .then(data => setPosts(data));       // 받은 데이터를 화면에 저장
}, []);
2
백엔드 (Express) — 요청 받아 DB 조회
"/posts" 경로로 요청이 오면 PostgreSQL에 SQL을 날린다 
// ── DB 연결 설정 (서버 시작할 때 한 번만) ──────────────────
const { Pool } = require('pg');         // pg = Node.js용 PostgreSQL 드라이버 (npm install pg)

const db = new Pool({                   // Pool = DB 연결을 미리 여러 개 만들어두는 객체
  host: 'localhost',                    // DB 서버 주소 (배포 시 실제 서버 IP)
  port: 5432,                           // PostgreSQL 기본 포트
  database: 'myapp',                    // 사용할 DB 이름
  user: 'postgres',                     // DB 계정
  password: 'secret',                   // DB 비밀번호 (실제론 환경변수로 숨김)
});
// 이제 db.query()를 쓰면 PostgreSQL에 SQL을 보낼 수 있음

// ── 라우터 ────────────────────────────────────────────────
app.get('/posts', async (req, res) => { // GET /posts 요청이 오면 이 함수 실행
  const result = await db.query(        // db.query() = PostgreSQL에 SQL 문장을 전송
    'SELECT * FROM posts ORDER BY created_at DESC'
    //  ↑ "posts 테이블에서 전체 행을 가져오되 최신 순으로 정렬해줘"
  );
  // result.rows = DB가 돌려준 행(row) 배열
  // 예: [{id:1, title:"첫글"}, {id:2, title:"두번째글"}, ...]
  res.json(result.rows);               // 그 배열을 JSON 형태로 프론트에 응답
});
흐름 요약: pg 드라이버가 Node.js와 PostgreSQL 사이 통역사 역할. db.query(SQL) 하면 드라이버가 네트워크로 PostgreSQL에 SQL 전달 → PostgreSQL이 데이터 꺼내서 돌려줌 → result.rows에 담김 → 프론트로 JSON 전송.
3
DB (PostgreSQL) — 데이터 반환
SQL 받아서 저장된 데이터 꺼내 돌려줌 
-- PostgreSQL이 실행하는 SQL
SELECT * FROM posts ORDER BY created_at DESC;
-- 결과: [{id:1, title:"첫글", content:"..."}, ...]
4
프론트 — 받은 데이터로 화면 그리기
JSON 데이터를 받아 React가 화면을 자동으로 업데이트 
// 받은 posts 배열을 화면에 표시
return (
  <ul>
    {posts.map(post => <li key={post.id}>{post.title}</li>)}
  </ul>
);

CRUD 4가지 — 글쓰기·보기·수정·삭제도 같은 방식

동작프론트가 보내는 것Express 라우터SQL
글 목록 보기GET /postsapp.get('/posts')SELECT * FROM posts
글 하나 보기GET /posts/3app.get('/posts/:id')SELECT * FROM posts WHERE id=3
글 쓰기POST /posts + 내용app.post('/posts')INSERT INTO posts ...
글 수정PUT /posts/3 + 내용app.put('/posts/:id')UPDATE posts SET ... WHERE id=3
글 삭제DELETE /posts/3app.delete('/posts/:id')DELETE FROM posts WHERE id=3
핵심 한 줄: 프론트는 "달라·줘·바꿔·지워"를 HTTP로 요청 → 서버는 받아서 DB에 SQL 실행 → 결과를 JSON으로 응답 → 프론트가 화면 업데이트. 이 한 바퀴가 웹 서비스의 전부다.

5. 권장 학습 순서

1
HTML·CSS·JS (프론트 기초) — 화면 먼저 만들어 보기
2
Python 기초 — 쉬운 언어로 프로그래밍 감 잡기
3
PostgreSQL + SQL — 데이터 다루기
4
FastAPI — Python으로 첫 API 서버 (게시판 CRUD 만들기)
5
Node.js + Express — 같은 게시판을 JS로 다시 (비교 학습)
6
TypeScript → React — 프론트 본격화, 만든 API와 연결
핵심: 각 단계마다 작은 프로젝트로 직접. "게시판 CRUD"를 FastAPI로 한 번, Express로 한 번 만들면 백엔드 핵심(라우팅·DB·인증)이 두 언어로 체화됨. (vault "생존법": 하나 깊게 → 둘째는 빠름)
