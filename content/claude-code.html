클로드 코드 — 명령어·개념·하네스 설계

코딩 AI 에이전트 도구. 개념 + 쉬운 설명 + 하네스 워크플로우 설계법.

① 슬래시 명령어 (/) — 공식 카테고리별

채팅창에 / 치면 전부 보임. 89개 중 자주 쓰는 것 분류(공식 docs 기준).

프로젝트 셋업

명령뜻 쉬운 예시 (언제·어떻게)
/initCLAUDE.md 자동 생성코드가 이미 있는 폴더에서 한 번 클로드가 그 코드 분석해 규칙파일 작성 (빈 폴더 X)
/memory규칙·기억 파일 편집"우린 항상 한국어로 답하고, 들여쓰기는 스페이스 2칸" 같은 규칙을 적어두면 매번 안 말해도 됨
/agents서브에이전트 관리"코드 리뷰 전담 부하 AI" 같은 걸 만들어 잡일 위임
/mcp외부도구 연결GitHub·Gmail·DB를 클로드에 꽂아 직접 다루게
/plugin플러그인 설치"슈퍼파워스 플러그인 깔아줘" 기능 묶음 통째 추가
/hooks자동 규칙 설정"파일 저장할 때마다 자동 포맷팅" 같은 자동 동작
/permissions도구 허용/차단"파일 삭제는 매번 나한테 물어봐" 안전장치
/config설정(테마·모델)다크모드·기본 모델 바꾸기
/add-dir작업 폴더 추가다른 폴더 파일도 보게 "이 폴더도 접근 허용"

작업 중 (계획·목표·모델)

명령뜻 쉬운 예시
/plan계획 먼저 모드큰 기능 만들기 전 "바로 코딩 말고 계획부터 짜서 보여줘" 검토 후 진행
/goal목표 달성까지 계속/goal 모든 테스트 통과 테스트 다 초록불 될 때까지 클로드가 안 멈추고 고침
/loop주기마다 반복/loop 5분마다 배포 상태 확인 5분마다 자동 점검(순찰)
/model모델 바꾸기어려운 일은 Opus, 간단한 건 Sonnet으로 전환
/effort /fast깊게 / 빠르게복잡하면 effort 올려 더 신중히, 급하면 fast로 빨리

컨텍스트(대화 기억) 관리

명령뜻 쉬운 예시
/context기억 용량 확인"지금 대화창 얼마나 찼지?" 뭐가 자리 차지하는지 보임
/compact대화 압축긴 작업 후 대화가 무거우면 "요약해서 가볍게" 핵심만 남김
/clear대화 비우기완전 다른 일 시작할 때 "이전 대화 다 지우고 새로"
/btw가벼운 곁다리"참고로 이거 궁금한데" — 본 작업 기억 안 늘리고 잠깐 물어보기
/recap여기까지 요약"지금까지 뭐 했지?" 작업 정리해줌

병렬·백그라운드 (여러 일 동시에)

명령뜻 쉬운 예시

/fork대화 복제지금까지 맥락 그대로 두고 "다른 방향도 시험" 갈래 치기

명령뜻 쉬운 예시

특수 입력 기호

! 명령그 줄을 셸 명령으로 실행 (예: !git status)
@ 파일파일 첨부·참조 (예: @src/app.js)

쉽게: /=리모컨 버튼, !=직접 터미널, @=파일 끌어다 놓기, #=AI한테 "이거 기억해".

①-2 터미널 CLI 기초 (클로드 말고 셸)

클로드 코드는 터미널에서 돌아감. 터미널 기본 명령부터 알아야 길을 안다. (Mac/Linux 기준)

폴더·파일 이동·보기

명령어뜻예
pwd지금 어디(현재 경로)pwd /Users/me/project
ls이 폴더 목록ls -la (숨김·상세까지)
cd폴더 이동(change directory)cd src / cd ..(상위) / cd ~(홈)
mkdir폴더 생성mkdir 새폴더
touch빈 파일 생성touch app.js
cat파일 내용 보기cat README.md
cp / mv복사 / 이동·이름변경cp a.txt b.txt / mv a.txt 폴더/
rm삭제( 복구 안 됨)rm a.txt / rm -rf 폴더(폴더째)
쉽게: 터미널 = 마우스 없이 키보드로 폴더 탐색. cd=폴더 들어가기, ls=뭐 있나 보기, pwd=나 지금 어디. 이 3개가 90%.

② 핵심 개념 6가지

CLAUDE.md

프로젝트 설명서·규칙 파일. 클로드가 작업 시작 때 항상 자동으로 읽음. "이 프로젝트는 이렇게 일해" 적어둠.

쉽게: 신입에게 주는 회사 매뉴얼. 매번 말 안 해도 알아서 봄. 가장 중요 — 핵심 규칙은 여기 둠.

스킬 (Skill)

.claude/skills/이름/SKILL.md. 특정 작업용 매뉴얼. 설명(description)이 맞으면 AI가 알아서 꺼내 씀.

예문: "휴머나이저 스킬" = AI 문체 제거. 글 다듬을 때 자동 발동.
쉽게: 책장에 꽂힌 전문 매뉴얼. 필요할 때 꺼냄. (단 AI가 안 꺼낼 때도 있어 핵심은 CLAUDE.md에).

서브에이전트 (Agent)

.claude/agents/이름.md. 일을 나눠 맡는 보조 AI. 자기만의 별도 대화창(컨텍스트)에서 작업 → 메인 오염 방지.

예문: "코드 탐색 에이전트"가 따로 돌며 파일 뒤져 결과만 보고 → 메인 컨텍스트 안 채움.
쉽게: 부하 직원. 메인(나)은 지시만, 잡일은 서브에이전트가 따로 처리 후 요약 보고.
스킬 vs 에이전트 — 헷갈리는 둘
스킬 (Skill)에이전트 (Subagent)
정체방법 매뉴얼 (지식)일하는 주체 (일꾼)
형태SKILL.md 글별도 컨텍스트(대화창)
컨텍스트메인에 주입(필요 시 로드)자기만의 격리 컨텍스트
실행스스로 일 안 함 — 읽혀서 따라짐도구 써서 실제 작업
질문어떻게 (방법)누가 (주체)
"역할만 나누고 일은 스킬로?" → 반만 맞음. 일은 에이전트가 함(자기 컨텍스트서 파일 읽고 코드 씀). 스킬은 그 일의 방법만 알려줌(스스로 실행 X). 한 에이전트가 여러 스킬 사용 가능, 스킬은 에이전트 없이 메인서도 쓰임.
쉽게: 에이전트 = 직원(몸), 스킬 = 그 직원이 보는 업무 매뉴얼(머리 속 방법). 직원이 매뉴얼 보고 일함.

흐름 예시 — "블로그 글 써줘"

① 메인: "블로그 글 써줘" 받음
↓ 무거운 조사 작업이라 분리
② 서브에이전트(작가) 호출 — 별도 컨텍스트서 일 시작
↓ 그 에이전트가 "글쓰기 스킬" 로드
③ 스킬(글쓰기 매뉴얼) 읽음 — "제목→서론→본문 3개→결론, 이 문체로" 방법 주입
↓ 매뉴얼대로
④ 에이전트가 실제 작성(read 자료 → write 초안)
↓ 결과만 메인에 반환
⑤ 메인은 깨끗한 상태로 초안 받음 (조사 과정으로 안 더럽혀짐)
한 줄: 에이전트(작가)가 몸으로 일하고, 스킬(글쓰기 매뉴얼)이 방법을 댄다. 둘이 합쳐 한 작업.

훅 (Hook)

settings.json에 정의. 특정 시점에 자동 실행되는 명령. AI가 아니라 시스템이 실행.

이벤트언제
SessionStart세션 시작 시
PreToolUse도구 쓰기 직전(차단 가능)
PostToolUse도구 쓴 직후
UserPromptSubmit사용자 입력 시
Stop작업 끝날 때
예문: PreToolUse 훅으로 rm(삭제) 명령 차단 실수 방지.
쉽게: 자동 센서. "이 순간이 오면 자동으로 이거 해라". 규칙은 사람이 코드로.

플러그인 (Plugin)

스킬+명령어+에이전트+훅+MCP를 묶은 패키지. 마켓플레이스에서 설치.

예문: "슈퍼파워스" 플러그인 = 계획테스트리뷰 강제하는 스킬·명령 묶음.
쉽게: 앱(여러 기능 한 묶음). 링크 복사 "설치해줘" 하면 통째 들어옴.

MCP (Model Context Protocol)

외부 도구·데이터에 연결하는 표준. 클로드가 GitHub·DB·Slack·드라이브 등과 대화하게.

// .mcp.json
{ "mcpServers": {
  "github": { "command": "npx", "args": ["mcp-github"] }
}}
쉽게: 만능 어댑터. 클로드에 외부 서비스(GitHub·Gmail 등)를 꽂아 직접 다루게. USB 포트 같은 표준.

③ 더 알아둘 관련 용어

용어뜻 / 쉽게
AGENTS.mdCLAUDE.md의 도구 중립 버전(표준화 움직임). 코덱스 등이 읽음. 클로드는 @AGENTS.md 임포트해야 봄 (↓ 우선순위 참고)
SOUL.md / IDENTITY.md표준 아닌 프로젝트 관습. 팀의 정체성·가치관을 적어 hook으로 항상 로드(이 위키가 쓰는 방식)
settings.json권한·환경변수·훅 설정 파일(.claude/)
permissions(권한)AI가 쓸 수 있는 도구 허용목록(Read·Write·Bash 등)
컨텍스트(context)AI가 기억하는 대화창. 차면 압축됨 아껴 써야
커스텀 커맨드 = 스킬공식: 커스텀 커맨드가 스킬로 통합됨. .claude/commands/deploy.md = .claude/skills/deploy/SKILL.md 둘 다 /deploy 생성(기존 commands도 계속 작동)
자동 메모리클로드가 작업하며 빌드명령·디버깅 교훈을 알아서 CLAUDE.md류에 저장(세션 넘어 유지)
오케스트레이션여러 에이전트의 흐름을 조율
permission mode자동승인 정도(plan·default·accept edits 등)

CLAUDE.md vs AGENTS.md — 둘 다 있으면 뭘 읽나

툴기본으로 읽는 파일
클로드 코드CLAUDE.md (AGENTS.md는 자동으로 안 읽음)
코덱스 등 타 툴AGENTS.md
둘 다 있으면? 클로드는 CLAUDE.md만 보고 AGENTS.md는 무시. 서로 폴백 안 함. AGENTS.md를 쓰려면 CLAUDE.md 안에서 @AGENTS.md로 임포트해야 함.
우선순위: CLAUDE.md에 직접 쓴 규칙 > 임포트한 AGENTS.md. 임포트는 가장 약한 층 — 충돌 시 직접 규칙이 항상 이김(작업 폴더에 가까운 게 뒤에 붙어 우선).
권장 패턴(Anthropic 공식): 공통 규칙은 AGENTS.md에 두고, CLAUDE.md엔 @AGENTS.md 한 줄 + 클로드 전용 오버라이드만. → 한 벌 규칙을 두 툴(클로드·코덱스)이 공유. 한 벌 관리

④ 멀티에이전트 시스템 (동적 워크플로우·팀즈·뷰)

서브에이전트 여럿을 굴리는 방식들. 구분 기준 = 조율 계획을 누가 드냐.

기능뭐특징
서브에이전트일 나눠 맡는 보조 AI목적 = 컨텍스트 격리(메인 안 채움)
동적 워크플로우메인이 스크립트(프로그램)를 만들어 서브에이전트를 실행·작업전달정적 조율 — 코드 재사용·통제 유리
에이전트 팀즈모델에 전부 위임("네가 다 해")동적 — 화려하나 결과 들쭉날쭉·통제 어려움
에이전트 뷰여러 에이전트 상태를 한눈에 보는 대시보드모니터링용
권고: 팀즈(완전 동적)는 매번 결과 달라짐 사람이 잘 만든 서브에이전트를 동적 워크플로우로 엮어 재사용하는 게 안정적. (이 위키도 정적 조율 채택)
쉽게: 메인 세션 = 소중한 작업 책상(차면 압축돼 기억 날아감) 잡일은 서브에이전트한테 따로 시켜 책상 아끼기. "야 알아서 다 해"(팀즈)보다 "이 순서로 해"(워크플로우)가 믿음직.

동적 워크플로우 — 공식 발표 (Anthropic)

핵심: 클로드가 오케스트레이션 스크립트를 그 자리에서 동적 생성 병렬 서브에이전트 실행 결과를 검증한 뒤 사용자에게 제출. 단일 에이전트가 못 할 만큼 복잡한 작업용.
항목내용
실증 사례Bun 프로젝트 — Rust 코드 약 75만 줄을 11일에 포팅, 테스트 99.8% 통과
작동큰 작업을 하위작업으로 분해 독립 관점의 에이전트 병렬 오류 비용 크면 대립 검증(adversarial)
활용처대규모 마이그레이션 · 코드베이스 전체 버그헌팅 · 보안 감사 · 최적화
가용성Max·Team 플랜 기본 활성 / Enterprise는 관리자 활성화
경고: 일반 세션보다 토큰 소모가 크게 늘어남 범위를 좁고 명확하게 정해 시작할 것. (= 우리 "투머치 하네스" 경계와 같은 결)
쉽게: "큰 공사를 인부 여러 명에 동시 분배 + 서로 교차검증". Bun은 이걸로 75만 줄을 11일 만에. 단 인부 많이 쓰니 비용(토큰) 폭증 — 딱 필요한 큰 일에만.

⑤ 하네스 구조 & 워크플로우 설계법

하네스 = AI 에이전트들을 팀으로 엮어 일 시키는 틀. 이 위키 팀이 살아있는 예시.

핵심 구성 5

구성역할이 위키 예
오케스트레이터전체 지휘(라우팅)CLAUDE.md
에이전트들각자 한 가지 일15개(문지기·요약가·검수…)
IO 계약고정 파일명으로 주고받기0_inbox2_areas 등
승인 게이트위험작업은 사람 승인삭제·분리 금지

워크플로우 설계 6원칙

1
목표·이해 먼저 — 뭘 달성할지부터. 도구·루프는 그다음.
2
일을 잘게 쪼개 1에이전트=1책임 — 한 명에 한 가지만.
3
고정 IO 계약 — 앞 단계가 약속된 곳에 쓰고 뒤가 읽음(동적 위임보다 정적이 통제·재사용 유리).
4
독립 검수 — 만든 에이전트가 자기 평가 금지. 별도 검수자(validator).
5
멈춤 장치 — 반복상한·정체감지·승인게이트(실수가 무인으로 쌓임 방지).
6
메모리·환류 — 작업 기록(run_log·knowledge) 다시 읽어 개선.
투머치 하네스 경계: 단계를 과하게 쪼개면 토큰 낭비·결과 산으로. "무엇을 넣느냐(컨텍스트)"가 단계 추가보다 중요. 핵심 규칙은 스킬(자동) 말고 CLAUDE.md(항상)에 + 압축. 선택을 시키면 틀린다.
쉽게: 하네스 = AI 직원 여러 명 둔 작은 회사 시스템. 지휘자(CLAUDE.md) + 직원들(에이전트) + 서류 규칙(IO 계약) + 결재(승인) + 일지(메모리). 단 직원·절차 너무 많으면 오히려 느려짐 — 딱 필요한 만큼.

⑥ 공식 기능 더보기 (docs.claude.com 기준)

기능뭐
번들 스킬기본 제공 명령: /debug(디버깅)·/code-review(코드리뷰) 등
예약 작업 3종Routines(/schedule, 앤트로픽 서버에서 — 컴퓨터 꺼져도 실행, GitHub 이벤트 트리거) / Desktop 예약(내 PC) / /loop(세션 내 폴링)
백그라운드 에이전트 + Agent View여러 세션 병렬 실행 + 한 화면에서 상태 보기(/en/agent-view)
헤드리스/파이프 모드claude -p "..." — 터미널 파이프·CI 자동화. 예: tail log | claude -p "이상하면 알려줘"
플랜 모드실행 전 계획 먼저 검토·승인(VS Code plan review)

멀티 서피스같은 엔진이 터미널·VS Code·JetBrains·데스크탑앱·웹(claude.ai/code)·iOS에서 — CLAUDE.md·설정·MCP 공유
외부 연동GitHub Actions(PR 자동리뷰)·GitLab CI·Slack(@ClaudePR)·Chrome(웹앱 디버그)·Channels(Telegram/Discord)
Teleport / Remote웹·모바일서 시작한 작업을 터미널로 끌어오기(claude --teleport), 폰서 원격제어
핵심 정정: ① 커스텀 커맨드 = 이제 스킬로 통합(둘 다 /이름 생성). ② 스킬 본문은 쓸 때만 로드돼 평소 컨텍스트 비용 거의 0 — 긴 참고자료는 스킬, 항상 필요한 사실은 CLAUDE.md. ③ 스킬은 Agent Skills 오픈표준(agentskills.io) — 다른 AI 도구와 호환.
쉽게: 클로드코드 = 터미널뿐 아니라 IDE·앱·웹·슬랙 어디서나 같은 두뇌. 예약(/schedule)으로 자는 동안도 일하고, -p로 파이프 자동화, SDK로 내 에이전트까지 만든다.

⑦ 예시 하네스 설계 + 파일 흐름

가정: "기능 개발 하네스" — 요청 한 줄 계획코딩테스트리뷰까지 자동. 클로드/코덱스가 파일을 어떻게 읽고 다음으로 넘기는지 그림.

폴더 구조 (하네스)

my-harness/
├ CLAUDE.md              # 규칙·라우팅 (항상 읽힘)
├ .claude/
│   ├ settings.json      # hook (SessionStart·PreToolUse)
│   ├ agents/            # 일꾼 4명 (각자 역할서)
│   │   planner.md  coder.md  tester.md  reviewer.md
│   └ skills/            # 작업 매뉴얼
└ work/                  # 데이터가 흐르는 곳 (IO 계약)
    ├ request.md         # ① 입력 (사람)
    ├ plan.md            # ② planner 산출
    ├ src/               # ③ coder 산출
    ├ test_result.md     # ④ tester 산출
    └ review.md          # ⑤ reviewer 산출

워크플로우 — 읽기()처리쓰기()

세션 시작 — hook이 CLAUDE.md+에이전트 역할서를 항상 자동 로드 (외운 방식 말고 이 규칙 먼저)
사람: "로그인 기능 만들어" work/request.md에 기록
오케스트레이터가 planner 호출
planner  read request.md 계획 수립 write plan.md
(plan.md가 다음 단계 입력 = IO 계약)
⌨ coder  read plan.md 코드 작성 write work/src/*
tester  read src/ 테스트 실행 write test_result.md
reviewer  read src/+test_result.md 검수 write review.md
승인 게이트
사람 승인 통과면 완료 / 실패면 planner로 되돌림(루프)

핵심 = 파일이 "바통"이다 (IO 계약)

원리설명
고정 파일명앞 단계가 plan.md에 쓰면, 뒤 단계가 그 이름으로 읽음. 약속된 경로 = 결합 지점
컨텍스트 격리각 에이전트는 자기 대화창에서 작업 메인 컨텍스트 안 더럽힘. 결과만 파일로 넘김
읽기쓰기 단방향전 단계 산출을 읽고, 다음 산출을 씀. 거꾸로 안 감(실패 시 루프로만 복귀)
독립 검수coder가 자기 코드 검수 X tester·reviewer가 따로(자기평가는 칭찬만 함)
예문 (클로드/코덱스 입장):
coder 에이전트는 명령받으면 Read work/plan.md(전 단계 산출 읽기) 내용 이해 Write work/src/login.js(코드 생성) "완료" 보고. plan.md를 안 만들었으면 coder는 뭘 할지 모름 — 그래서 순서·파일명이 계약.
쉽게: 공장 컨베이어 벨트. 1번이 만든 반제품(plan.md)을 2번이 받아 가공(src)하고 3번에 넘김. 각자 자기 자리에서 앞 상자 열어보고(read) 다음 상자에 담기(write). 상자 이름이 정해져 있어 안 헷갈림.
이 위키 팀이 바로 이 구조: 0_inboxcapture_routerdistiller2_areas. 파일명이 고정 IO 계약, 15 에이전트가 바통터치, 사람 승인 게이트, 실패는 루프 복귀.
